# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Laminar is an open-source observability platform for AI agents. It provides OpenTelemetry-native tracing, evaluations, AI monitoring, and SQL access to all data.

## Repository Structure

This is a multi-service monorepo with three main components:

- **app-server/** - Rust backend (Actix-web HTTP, Tonic gRPC)
- **frontend/** - Next.js/TypeScript web UI
- **query-engine/** - Python gRPC service for SQL query processing
- **pii-redactor/** - optional Rust gRPC service that runs a HuggingFace token-classification PII model on CPU via ONNX Runtime. Standalone ÔÇö not linked from app-server. Tested with the OpenAI privacy filter (BIOES) and Piiranha (BIO); accepts either scheme via `config.json` `id2label`. See `pii-redactor/README.md` for the gRPC contract, model layout (`model.onnx` + optional `model.onnx_data*` external-data shards + `tokenizer.json` + `config.json`), and the weight-baking Dockerfile.

## Database Migrations

Database schema is managed with Drizzle ORM. The source of truth is the database itself - do NOT edit schema files directly.

```bash
cd frontend
pnpm db:generate                # Generate migrations AND strip "public". qualifiers (preferred)
# or, equivalently:
npx drizzle-kit generate && pnpm db:strip-schema
# Migrations are applied automatically on frontend startup
```

- `pnpm schema-pull:lint` heavily reformats `schema.ts`, `relations.ts`, and `tsconfig.json`. After running it, revert unrelated formatting changes before committing.
- **Migration SQL must stay schema-neutral (no `"public".` qualifiers).** Tables resolve via the connection `search_path` (`POSTGRES_SCHEMA`, default `public`), so a hardcoded `"public".` would break non-public self-hosted installs. `drizzle-kit generate` re-emits `"public".` prefixes (it has no schema-neutral output mode) ÔÇö NOT only on tables, but also on **foreign-key targets** (`REFERENCES "public"."projects"(...)`), **enum/type creation** (`CREATE TYPE "public"."event_source" ...`), and **column type changes** (`SET DATA TYPE "public"."event_source"`). `pnpm db:generate` runs the generate then `pnpm db:strip-schema`, which `sed`s every `"public".` token out of all migration `.sql` files. If you generate by hand, run `pnpm db:strip-schema` (or `sed -i 's/"public"\.//g' <file>`) afterwards. The strip targets only the `"public".` schema qualifier (double-quoted name + dot), so it leaves `'public'` string literals, `public_`-prefixed constraint names, and `public_key` columns intact. Snapshots/`_journal.json` already record an empty schema, so they need no change.
- `npx drizzle-kit generate` requires a TTY for interactive prompts. In non-interactive shells (CI, sandbox), write migration SQL files and `_journal.json` entries manually.
- When writing migrations manually, also create a `meta/NNNN_snapshot.json`. Copy the previous snapshot, apply the schema change (e.g. add/remove columns), set `prevId` to the previous snapshot's `id`, and generate a new UUID for `id`. Without a snapshot, the next `drizzle-kit generate` will produce a duplicate migration.
- **ClickHouse migrations** (`frontend/lib/clickhouse/migrations/`) are tracked by the migration tool and only run once. Never modify an already-applied migration file ÔÇö changes won't execute on existing deployments and may cause checksum errors. Always create a new numbered migration file instead.

## Configurable Postgres schema (`POSTGRES_SCHEMA`)

- `POSTGRES_SCHEMA` (default `public`) is the schema all Postgres tables live in. It's applied as the connection `search_path` in BOTH services ÔÇö frontend `lib/db/drizzle.ts` (`connection: { search_path }` on the `postgres()` client) and app-server `db/mod.rs` (`PgConnectOptions::options([("search_path", ÔÇŽ)])`, descriptor `env::database::SCHEMA`). All queries use unqualified table names, so the search_path is the only routing mechanism; the two services MUST be set to the same value.
- The frontend resolves the value via `getPostgresSchema()` in `drizzle.ts`: empty/unset means "no explicit schema" (connection defaults to `public`). For the connection `search_path`, any explicit value ÔÇö INCLUDING `public` ÔÇö is applied. For the migrations-tracker location (`migrationsSchema`) and the boot-time `CREATE SCHEMA`, any case variant of `public` is treated like unset (the check is `postgresSchema.toLowerCase() === "public"`).
- **`migrationsSchema` guard (instrumentation.ts):** `migrate()` is passed `migrationsSchema: <schema>` ONLY when an explicit `POSTGRES_SCHEMA` is set AND it is not a case-insensitive `public`. This is deliberate: relocating `__drizzle_migrations` into a schema where it doesn't already exist makes the migrator see "no last migration" and re-run all migrations (non-idempotent `ALTER TYPE ÔÇŽ ADD VALUE` / `DROP CONSTRAINT` then errors). Existing default deployments ÔÇö whether `POSTGRES_SCHEMA` is unset OR explicitly `public`/`PUBLIC` ÔÇö have their tracker in `drizzle` and must stay there, so an operator who sets `POSTGRES_SCHEMA=public` "to be explicit" doesn't trip a re-run. So: unset or `public` (any case) Ôćĺ tracker stays in `drizzle`; any other explicit schema Ôćĺ tracker moves into that schema (lets a Laminar DB coexist with another Drizzle service in the same instance).
- `POSTGRES_CREATE_SCHEMA` (default `true`, frontend-only) gates the boot-time `CREATE SCHEMA IF NOT EXISTS`. Set `false` when the schema is pre-provisioned or the DB role lacks `CREATE`. The data schema is NOT auto-created by the migrator ÔÇö only the migrations-table schema is ÔÇö so this DDL is required for a fresh non-public install. The `CREATE SCHEMA` is skipped for any case variant of `public` (it always exists, and quoting `"PUBLIC"` into the DDL would create a distinct schema that the lowercase-folding `search_path` never resolves to).

## SQL Editor Schema (autocomplete + AI generation)

- `frontend/components/sql/utils.ts` (`tableSchemas`, `enumValues`) is the frontend source of truth for the SQL editor's autocomplete AND for the schema embedded in the AI-generation system prompt (`frontend/lib/actions/sql/prompts.ts` consumes both). Backend allowlist lives in `query-engine/src/query_validator.py` Ôćĺ `TableRegistry._setup_default_tables`; the two MUST stay in sync ÔÇö adding a column to the v0 view + the registry without updating `tableSchemas` leaves it absent from autocomplete and invisible to the AI generator.
- Columns whose runtime CH type is `String` but whose values are constrained (status, span_type, trace_type, signal_runs.status / .mode) carry an `enumType: keyof typeof enumValues` field. The prompt builder appends `Allowed values: '...'` to the column's description, and the editor's `<column> = ` autocomplete filters to those literals. Without `enumType`, the model invents values (the original LAM-1636 bug: AI generated `status = 'OK'` because nothing pinned the status column to `'success' | 'error'`).
- `enumValues` is keyed by enum NAME, not column name. A single column can map to one enum (`spans.status` Ôćĺ `status`) or different enums in different tables (`signal_runs.status` Ôćĺ `signal_run_status`). The autocomplete `columnEnumMap` is built by walking `tableSchemas` once at module load, so columnÔćĺenum wiring stays declarative ÔÇö never add column names to a hardcoded regex; just set `enumType` on the schema entry.
- The `clusters_v0` view (CH migration; latest is `47_clusters_v0_drop_centroid.sql`) intentionally does NOT expose `centroid` ÔÇö the embedding vector is an internal `Array(BFloat16)` on `signal_event_clusters`, useless over SQL and expensive to ship. When recreating the view in a new migration, keep the `WHERE level > 0` L0 filter and the 9 user-facing columns (id, signal_id, name, level, parent_id, num_signal_events, num_children_clusters, created_at, updated_at). `clusters` is now in `tableSchemas` (autocomplete + AI + docs). `signal_events_all` stays registry-only (queryable) and is intentionally NOT in `tableSchemas`.

## Labeling Queue Items (ClickHouse)

- Queue metadata (`labeling_queues`) lives in Postgres. Per-item rows (`labeling_queue_items`) live in ClickHouse as `ReplacingMergeTree(updated_at)` keyed `(project_id, queue_id, id)` ÔÇö see `frontend/lib/clickhouse/migrations/42_labeling_queue_items.sql`, which also creates the `labeling_queue_items_v0` view (registered in `query-engine/src/query_validator.py`).
- Low-level CH helpers (`insertQueueItems` / `getQueueItems` / `getQueueItemStates` / `updateQueueItem` / `deleteQueueItems` / `deleteQueueItemsByQueueIds` / `findQueueIdsByProgress` / `getQueueProgresses` / `copyQueueItemsToDataset`) live in `frontend/lib/actions/queue/items.ts`, co-located with the actions in `index.ts`. `lib/clickhouse/` is for cross-feature primitives (client, migrations) only.
- **Column model (mirror ÔÇö `edit` is always the canonical current target):** `payload` is **immutable** after insert (the seeded `{data, target, metadata}` snapshot). The `edit String` column (no SQL `DEFAULT`; every insert writes it explicitly) carries the current target as a JSON string, seeded equal to `payload.target` on insert (Rust `create_labeling_queues_items`, TS `pushQueueItems`) and overwritten by every UI PATCH. `status UInt8` (no SQL `DEFAULT`) replaces the old `is_labelled Bool` (0 = unlabeled, 1 = approved; `u8` leaves room for future states). The v0 view exposes `edit AS edit` (NO rename to `target`, no coalesce); callers resolve the friendly `target` themselves via `getEffectiveTarget` (`frontend/components/queue/queue-store/helpers.ts`) / `effectiveTarget` (`frontend/lib/actions/queue/index.ts`), which parse `edit` and fall back to `payload.target` only defensively for legacy rows.
- **Why mirror, not sentinel:** seeding `edit` everywhere means `dirty` is a value compare, not a write-detector ÔÇö reverting an edit to the original correctly drops the dirty flag, which a sentinel (`edit !== ""`) can't. `isDirty` deep-equals `JSON.parse(edit)` vs `payload.target` via `lodash.isEqual`, both normalised to `null` first (an omitted target parses to `undefined` but `edit` is `"null"`, and `isEqual(null, undefined)` is false). `isApproved = status === 1`. Dirty is computed only inside the loaded window (the index query never returns `payload`/`edit`).
- **All reads use `FINAL`**. Writes append; no `UPDATE`. The RMW helper `updateQueueItem` re-inserts with fresh `updated_at`, preserves immutable fields (`payload`, `metadata`, `createdAt`, `idempotency_key`), writes only `edit`/`status`. Its narrow surface `{ id, queueId, projectId, edit?, status? }` deliberately omits `payload`; never pass `edit: ""`.
- Idempotency = **deterministic id derivation**, not a pre-flight check. With an `idempotencyKey`, the handler derives `id = Uuid::new_v5(&queue_id, idempotency_key.as_bytes())`; a retry lands on the same sort-key tuple and collapses on FINAL (last write wins by `updated_at`). No key Ôçĺ `Uuid::now_v7()`. Same-key duplicates within one batch are deduped via a `seen_keys` HashSet. This replaced a lookup-then-insert that had a concurrent-insert race window. **Trade-off:** `payload` is not strictly immutable across retries ÔÇö a same-key retry arriving after a PATCH/approve reverts both columns to seed values (larger `updated_at`). Retries fire seconds after ingest (before any human edit), so this is rare. Frontend PATCH RMW still preserves `payload` (it copies it forward); only public-API retries can rewrite it.
- `insertQueueItems` opts OUT of `async_insert` (`async_insert: 0`). With async inserts, `wait_for_async_insert` acks at buffer-entry (flushed ~200 ms), so a rapid follow-up RMW could FINAL-SELECT before flush, see nothing, and re-insert with empty `edit` (catastrophic under the mirror model). Sync inserts make writes immediately visible. The table is low-volume, so no throughput loss.
- **Public ingest API does NOT accept `edit`.** `LabelingQueueItemRequest` takes `{ data, target, metadata, idempotencyKey }`; the handler seeds `edit = serde_json::to_string(&target)`, `status = 0`. Overwrites flow only through the frontend PATCH (`/api/projects/<id>/queues/<id>/items/<id>` Ôćĺ `updateQueueItemEdit` Ôćĺ `updateQueueItem`).
- Deletes use CH **lightweight DELETE** (tombstone FINAL respects). `deleteQueueItems` / `deleteQueueItemsByQueueIds` wrap it. On `deleteQueues`, call `deleteQueueItemsByQueueIds` BEFORE the Postgres DELETE ÔÇö FK cascades only cover Postgres.
- `pushItemsToDataset` defaults to pushing only `status = 1` rows; `includeUnlabelled: true` pushes all (the "All items in queue" radio in `push-to-dataset-dialog.tsx`). It returns `{ pushed: <count> }` with **200 even when nothing was pushed**, so gate local removal / success toasts on `result.pushed > 0`, not `res.ok`. Both push and `removeQueueItem` (DELETE-with-`datasetId`) go through `copyQueueItemsToDataset`, which copies server-side via `INSERT INTO dataset_datapoints SELECT ÔÇŽ FROM labeling_queue_items FINAL` (`edit` Ôćĺ `target`). Datapoint ids are minted in JS (`generateSequentialUuidsV7`) and zipped positionally with `row_number() OVER (ORDER BY id)` + `arrayElement` (the old `indexOf`-zip collapsed to the zero UUID on format drift). A post-INSERT verify `SELECT count() WHERE id IN newIds` throws on mismatch; only then does `deleteQueueItems` run. Ids are non-deterministic and `dataset_datapoints` is plain `MergeTree`, so a verify-passed-but-delete-failed retry duplicates ÔÇö the UI gates the push button in-flight, so it's acceptable.
- Edit auto-save is the closure-scoped factory `createSaveOrchestrator` (`queue-store/save-orchestrator.ts`, used from `index.tsx`), tracking timers/abort controllers **per-item-id** in `Map`s. Do NOT collapse to a single `useDebounce(currentItem)` ÔÇö a shared timer is cleared on nav within the 600 ms window, dropping A's edit if the user tabs to B. Surface: `schedule` / `cancel` / `cancelAll` / `flushAllPending` / `hasPending`. On `QueueDataLoader` unmount, `flushPendingSaves` Ôćĺ `flushAllPending` fires all pending PATCHes. Each debounced save sends `{ edit, status: 0 }` (editing reverts approval).
- Approve / unapprove / discard / push must call `orchestrator.cancel(id)` first ÔÇö else a delayed `status: 0` PATCH lands after approve's `status: 1` and RMT last-write-wins reverts it (same for discard resurrecting past a tombstone). `pushAllToDataset` calls `cancelAll()` (windowing means the client can't enumerate all ids). Approve/unapprove bodies are `{ status }` only ÔÇö no target (canonical value lives in `edit`).
- **Lazy windowed loading**: state split into `idsList: string[]` + `itemStates: Record<id, 'new'|'modified'|'approved'>` (full ordering + lifecycle, hydrated once via `getQueueItemStates`) + `loadedItems` (sparse, current window only). `WINDOW_RADIUS = 2` (ÔëĄ5 bodies in memory). `setCurrentIndex`/`step` run `evictOutsideWindow` but skip any id with a pending save (`hasPending`) so the trailing PATCH never fires against evicted state.
- `getQueueItems` accepts opt-in `ids?: string[]`; empty `ids: []` short-circuits to `[]` (avoids `IN ()` CH syntax error). `limit`/`offset` are also opt-in (no `LIMIT` when unset).
- The `/api/projects/.../queues/.../items` route is **mode-multiplexed**: `?ids=<csv>` Ôćĺ `{ items }` via `listQueueItems` (window); no `ids` Ôćĺ `{ items: [{id, state}] }` via `listQueueItemStates` (one tuple per row in `(created_at, id)` order, no `payload`/`edit` bytes). Both branches return an `items` key. The index response is SWR-cached by `QueueDataLoader`; window fetches use a manual `fetch` + `AbortController`. `state` derives in SQL via `multiIf` on `(status, edit, JSONExtractRaw(payload, 'target'))`; the `new` vs `modified` split is a JSON-string compare on raw `edit` (stable for value-only edits since both Rust insert and TS PATCH preserve key order). Mirrored in frontend `deriveItemState` for optimistic repaints.
- Navigator bar (`frontend/components/queue/navigator-bar.tsx`) renders one segment per item off `idsList` + `itemStates` (muted/amber/green). Legend counts come from `computeProgress` kept as a `progress` state field (recount only on state changes). Replaces the old percentage + filter-chips combo; there is no server-side `?statusFilter`. `approveCurrent`/`unapproveCurrent` flip `loadedItems[id].status` + `itemStates[id]` + recompute progress in one setState; `approveCurrent` auto-steps next (no filter to collapse the index).
- Queue hotkeys (`frontend/components/queue/hotkeys.tsx`) are scoped **per-shortcut**: `ÔîśÔĆÄ` (approve/unapprove) uses `{ enableOnFormTags: true, enableOnContentEditable: true }` so it fires inside CodeMirror/inputs; `ÔîśÔîź` (discard), `ÔîśÔćÉ`/`ÔîśÔćĺ` (prev/next) use the **default scope** so they don't fire while editing (else `ÔîśÔîź` discarded the item mid-JSON-edit). Press `Esc` to blur first. Do NOT merge into one shared options object.
- `created_at`/`updated_at` are `DateTime64(3, 'UTC')`. In Rust, model as `u64` millis (`Utc::now().timestamp_millis() as u64`); the TS client reads ISO strings via `formatDateTime(..., '%Y-%m-%dT%H:%i:%S.%fZ')`.
- `getQueues` (`frontend/lib/actions/queues/index.ts`) drops `annotationSchema` and joins per-queue counts via `getQueueProgresses` (one grouped CH query per page). Shape `LabelingQueueWithProgress = LabelingQueue + { progress }`.
- **Progress filters** (`total`/`new`/`modified`/`approved`) are cross-datastore: `getQueues` partitions `?filter=ÔÇŽ` via `isProgressFilter` ÔÇö progress ones go to `findQueueIdsByProgress` (CH `GROUP BY ÔÇŽ HAVING`) and the id set joins Postgres as `inArray(...)`; name/id stay in the PG `WHERE`. `PROGRESS_EXPR` mirrors `getQueueItemStates`/`deriveItemState`. **Limitation:** the GROUP BY only sees queues with Ôëą1 item, so `= 0` predicates miss empty queues (`approved = 0` = "has items but none approved"). Common filters work; for zero-including-empty, add an inverse `NOT IN` query rather than UNION-ing the empty set.

## Ingestion Rate Limiting

- Per-project rate limiting is driven by a single `actix_limitation::Limiter` built in `app-server/src/main.rs` when `Feature::RateLimiter` is enabled (`REDIS_URL` + `RATE_LIMIT` + `RATE_LIMIT_PERIOD_SECS`). HTTP wraps it via `actix_limitation::RateLimiter` middleware on the relevant scopes; gRPC ingestion (`ProcessTracesService::export` in `traces/grpc_service.rs`) calls `limiter.count(format!("ratelimit:{}", project_id))` inline after auth. Both paths use the same Redis key prefix, so OTLP/HTTP and OTLP/gRPC share one quota ÔÇö clients can't bypass the limit by switching transports.
- Tonic has no actix-style middleware; the limiter is plumbed into `ProcessTracesService::new` as `Option<Arc<Limiter>>` and the gRPC thread receives a clone (`grpc_rate_limiter = rate_limiter.as_ref().map(|l| Arc::new(l.clone()))`) **before** the HTTP closure moves `rate_limiter`. Don't reorder those two clones.
- Posture is fail-open on Redis errors (logged, request allowed) ÔÇö same as the bytes-limit check in the same handler ÔÇö so a Redis blip can't black-hole ingestion. `LimitExceeded` returns `Status::cancelled("Rate limit exceeded")` rather than `resource_exhausted`: OTel SDK retry policies treat `resource_exhausted` as retriable only when the server attaches `RetryInfo`, so `cancelled` is the safer "legitimate clients please retry" signal here.

## OTLP Trace Ingestion (`/v1/traces`)

- The HTTP handler dispatches on `Content-Type`: `application/json` decodes via `crate::opentelemetry_json` (serde shadow types Ôćĺ prost types), anything else falls through to `prost::Message::decode` (HTTP+protobuf, the default for every existing SDK). The gRPC server is unchanged. See `app-server/src/api/v1/traces.rs:decode_export_trace_request`.
- `build.rs` regenerates `src/opentelemetry_proto/mod.rs` from scratch every build via `tonic_prost_build::configure().include_file("mod.rs")`. Anything added inside `src/opentelemetry_proto/` will be clobbered on the next `cargo build`. Hand-written modules that work against the OTel proto types must live outside that directory (e.g. `src/opentelemetry_json.rs` declared from `main.rs`).
- The JSON decoder accepts the spec quirks browser SDKs actually emit: hex *or* base64 IDs, decimal-stringified `fixed64` *or* raw numbers, enum *names* (`SPAN_KIND_INTERNAL`) *or* ordinals for `kind` / `status.code`, and `intValue` as either string or number. Don't tighten any of these without checking real client output.

## App-server self-tracing (`src/instrumentation/`)

- This is **OTEL self-tracing infrastructure** ÔÇö the app-server tracing ITSELF, distinct from the `traces` module which ingests client spans. Annotate a function with `#[tracing::instrument(target = "lmnr::internal")]` and its OTEL span is ingested via an in-process exporter into a Laminar project, instead of threading a tracer/queue handle through every call site. Set attributes via `tracing_opentelemetry::OpenTelemetrySpanExt::set_attribute`; add `parent = None` to root a fresh internal trace.
- `setup_tracing_and_logging` runs TWO independent `tracing-opentelemetry` layers, each with its own `SdkTracerProvider`, so Sentry tracing and Laminar's internal self-tracing never share a trace_id or corrupt each other's span context. A single-provider "routing processor" design was rejected because it produced mixed trees (torn parent-child links). The two layers carry **disjoint per-layer filters** (`!is_internal` vs `is_internal`, where `is_internal` = `target` starts with `"lmnr::internal"`); each span is admitted by exactly one layer so the stock bridge layers never double-write `OtelData`. Do NOT add a third unfiltered `tracing_opentelemetry::layer()`.
- The **`fmt` (stderr) logger also carries `!is_internal`**, so internal-target events never print as application logs. This matters because attaching an `exception` event to an internal span is done by emitting an internal `tracing::error!` with an `error` field (`spans::record_error`) ÔÇö without the filter every such event would spam stderr. Same rule as `sentry_layer`: anything `lmnr::internal` stays out of the app logger AND out of Sentry.
- **`RUST_LOG` (`env_filter`) MUST be a per-layer filter on the `fmt`/Sentry layers, NOT `.with(env_filter)` on the registry.** A registry-level filter is GLOBAL: any layer whose `enabled` returns `false` short-circuits the span/event for the ENTIRE stack, so a module-scoped `RUST_LOG` like `app_server=debug` would silently starve the internal export layer. The default `EnvFilter::new("info")` masks the bug.
- The **Sentry provider stays the global tracer provider** (`global::set_tracer_provider`) and owns `SentrySpanProcessor`. The **internal provider is deliberately never set global** ÔÇö that plus the disjoint filter keeps internal spans out of Sentry and Sentry spans from parenting internal spans.
- The two trees are gated INDEPENDENTLY: the Sentry tree on `Feature::Tracing` (`SENTRY_DSN` + `ENABLE_TRACING`), the internal self-tracing tree on `Feature::InternalTracing` (`ENABLE_TRACING` only ÔÇö no Sentry DSN required). The internal tree deliberately does NOT depend on Sentry. When internal tracing is disabled the internal provider isn't built and internal spans drop Ôçĺ zero behaviour change. Each internal span carries its own destination project id via the `lmnr.internal.project_id` attribute (`INTERNAL_PROJECT_ID_ATTR`); the exporter routes per span and a `None` project id makes the span a no-op.
- **The internal exporter is in-process, NOT an OTLP loopback** (`internal_exporter.rs`, `InProcessInternalExporter`). It maps the SDK's `SpanData` onto the proto `ExportTraceServiceRequest` and calls `push_spans_to_queue` directly ÔÇö same ingest path OTLP/gRPC feeds, minus the network hop. So there's no internal project API key and no auth header.
- **Threading / deferred deps:** the batch span processor drives `SpanExporter::export` on its own non-tokio thread, so the exporter holds a tokio `Handle` and `block_on`s the async ingest on it. But `setup_tracing_and_logging` runs in `main.rs` BEFORE the queue/DB/cache exist, while the provider must be built there so the subscriber initialises once. The bridge is `SharedIngestDeps = Arc<OnceLock<IngestDeps>>`: `setup_tracing_and_logging` returns it alongside the provider and `main` calls `.set(...)` once those services exist (right after the `queue` binding). Until set, the exporter drops spans.
- `setup_tracing_and_logging` takes `(enable_sentry_tracing, enable_internal_tracing, &tokio::runtime::Handle)` and RETURNS `(Option<SdkTracerProvider>, SharedIngestDeps)`. The returned provider (the internal tracer) is `#[must_use]` and main must hold it (`let _internal_tracer_provider = ...`) for the process lifetime so the batch exporter keeps flushing ÔÇö drop it and internal spans stop exporting. Main must also populate the returned `SharedIngestDeps`.
- **Self-ingestion guardrail:** NEVER add `target = "lmnr::internal"` to the ingest path (`traces/grpc_service.rs`, `traces/producer.rs`/`push_spans_to_queue`, `db/spans.rs`, the span consumer). An internal span emitted from inside `push_spans_to_queue` would re-enter it and recurse.
- `instrumentation/spans.rs` is the generic internal-span builder (shared `lmnr.span.*` / `gen_ai.*` / `signal.*` / association-prop conventions, `lmnr.span.input`/`output` JSON-stringified via `json_attr`, `ai.prompt.tools` as an OTEL string array via `tools_attr`). It's `#[allow(dead_code)]` because no OSS producer consumes it yet ÔÇö the trace-chat agent that emits via this path is enterprise-only and lives in `lmnr-private`.

## OTel GenAI Semantic Convention Ingestion

- **Backend (app-server)** preserves the native OTel GenAI message shape end-to-end ÔÇö `gen_ai.input.messages` / `gen_ai.output.messages` are deserialised from JSON string to `Value` but NOT reshaped into Laminar's `ChatMessage` struct. This keeps the original instrumentation payload lossless; the frontend owns rendering. Do NOT reintroduce a backend ChatMessage conversion here ÔÇö this was explicitly reverted in favour of keeping the raw shape.
- `gen_ai.system_instructions` is prepended to the input messages as a synthetic `{role: "system", parts: [...]}` entry so the system prompt threads into the same message array. Bare-string arrays (`["Be helpful"]`) are preserved verbatim inside `parts` ÔÇö the frontend parser handles both shapes. Helper: `prepend_system_instructions` in `spans.rs`.
- **Frontend parser** lives at `frontend/lib/spans/types/gen-ai.ts` (`parseGenAIMessages`). It runs in `processMessages` (`frontend/components/traces/span-view/messages.tsx`) AFTER the OpenAI/Anthropic/LangChain/Gemini detectors and decodes the GenAI parts (`text|thinking|tool_call|tool_call_response|uri|blob`) into `ModelMessage[]` for the existing generic renderer.
- Span-type inference from `gen_ai.operation.name`: `chat|text_completion|embeddings|generate_content` Ôćĺ LLM, `execute_tool` Ôćĺ Tool, `invoke_agent` Ôćĺ Default.
- **pydantic_ai tool-span quirk**: pydantic_ai emits `execute_tool <name>` spans WITHOUT `gen_ai.operation.name`; only `gen_ai.tool.call.arguments` / `gen_ai.tool.call.result` are set. `span_type()` has a fallback that classifies a span as Tool when either of those attributes is present. Do not remove this fallback when adding other GenAI emitters.
- The `gen_ai.tool.call.*` Ôćĺ `self.input`/`self.output` block in `parse_and_enrich_attributes` is gated on `self.span_type == SpanType::Tool` (not just the raw attribute keys) so it cannot clobber LLM-span input/output if a spec-violating emitter mixes LLM message attrs with tool-call attrs on the same span. The gate still works for pydantic_ai because `span_type()` already infers Tool from the `gen_ai.tool.call.*` fallback before this runs.

## Billing and Tier Allowances

- Tier included allowances are hardcoded in two places that MUST stay in sync: frontend `TIER_CONFIG` in `frontend/lib/actions/checkout/types.ts` (`includedBytes` / `includedSignalSteps`) and Rust `WorkspaceTierName` methods in `app-server/src/db/projects.rs` (`included_bytes()` / `included_signal_steps()`). The backend uses these to detect whether a triggered usage warning corresponds to the tier's free allowance vs. a user-configured custom threshold ÔÇö this drives tier-specific copy in the usage-warning email ("you've exhausted your included Hobby allowance, from now on usage is billed pay-as-you-go").
- Stripe invoice line items expose their lookup key in two places: legacy `line.price.lookup_key` and the newer `line.pricing.price_details.price.lookup_key`. Email-subject/body rendering must check BOTH before falling back to `line.description` or a generic "Subscription charge" string, otherwise the subject line silently becomes `Laminar: Payment for  is received.` (trailing empty label).
- Transactional-email "billing portal" CTAs link to `/checkout/portal?workspaceId=...` (`frontend/app/checkout/portal/page.tsx`), which mints a Stripe billing portal session via `getPaymentMethodPortalUrl` and 302s straight to Stripe. Unauthenticated clicks bounce through `/sign-in?callbackUrl=...`. `getPaymentMethodPortalUrl` requires owner/admin role on the workspace ÔÇö non-admin recipients hit the catch and get redirected to the workspace billing tab as a graceful fallback. Don't link emails directly to the workspace settings page; that path requires the user to manually click "Billing portal" again.
## Workspace Usage Warnings vs Hard Limits

- Usage enforcement lives in two separate tables, both edited via Stripe webhook when a subscription is created or a tier is switched (`frontend/lib/actions/checkout/webhook.ts`):
  - `workspace_usage_warnings` ÔÇö soft limits that fire a notification once per billing cycle. Checked in `check_soft_limits` in `app-server/src/utils/limits.rs`.
  - `workspace_usage_limits` ÔÇö hard caps read by `get_project_and_workspace_billing_info` as `custom_bytes_limit` / `custom_signal_steps_limit` and enforced in `get_effective_*_limit` (app-server) and `checkSignalRunsLimit` (frontend).
- On tier transition into **Hobby**, we insert a soft warning row at the Hobby included allowance (5,000 signal steps) AND a hard-limit row for `signal_steps_processed` at `HOBBY_DEFAULT_HARD_LIMIT_SIGNAL_STEPS` (15,000). The hard cap is intentionally set to the same threshold as the overage email warning so ingestion is blocked at the same moment the user gets nudged. Pro and Free do NOT get a default hard limit.
- Hobby also gets two extra **overage warnings** above the included allowance (15,000 signal steps, 40 GiB bytes) so users accumulating a large overage bill are nudged before it grows further. These are Hobby-only rows; Pro does not get them. On transition OUT of Hobby they are cleared (matching exact value so user-adjusted thresholds are preserved) via `clearHobbyOverageWarnings`, which ÔÇö like the hard-limit cleanup ÔÇö runs on every transition including Hobby Ôćĺ Free.
- On tier transition OUT of Hobby, the default Hobby hard limit is deleted **only if it still equals a known Hobby default** ÔÇö either the current `HOBBY_DEFAULT_HARD_LIMIT_SIGNAL_STEPS` (15,000) or the legacy `currentTierConfig.includedSignalSteps` (5,000, for workspaces whose row was written before the cap was raised). A user-set custom value is preserved across upgrades/downgrades. Same pattern is applied to warnings.
- The hard-limit cleanup runs on **every** recognized tier transition ÔÇö including Hobby Ôćĺ Free on cancellation. Warnings, in contrast, are only written when the new tier is paid (Hobby/Pro) since Free has no warnings to carry. Leaving the Hobby limit in place during cancellation would silently re-apply the Hobby cap the next time the workspace upgrades to any paid tier.
- After inserting or deleting a `workspace_usage_limits` row, you MUST invalidate the per-project cache via `invalidateProjectCacheForWorkspace(workspaceId)` ÔÇö the app-server caches `ProjectWithWorkspaceBillingInfo` keyed by project id and will otherwise keep enforcing stale limits until TTL.

## AI Features Model Setup

- All frontend AI features (chat-with-trace, SQL generation, span previews, session extraction regex, debugger-session naming) go through `getLanguageModel(tier)` in `frontend/lib/ai/model.ts`. Tiers are `small` / `medium` / `large` ÔÇö never add new tier names without updating every call site.
- SQL "Ask AI" generation modes (`GenerationMode` in `frontend/lib/actions/sql/types.ts`: `query` / `eval-expression` / `trace-expression`) are coupled across THREE places that must stay in sync: the `GenerationMode` type, `MODE_CONFIGS` in `frontend/lib/actions/sql/prompts.ts`, AND the `mode` zod enum in `frontend/lib/actions/sql/generate.ts`. The generate action's enum is the validation gate ÔÇö a mode missing there throws `Invalid option: ... at mode` at request time even though the type and prompt config accept it (LAM-1679: traces-table custom-column AI gen sent `trace-expression`, which the enum didn't list). Each table's columns menu picks its mode (`traces-columns-menu` Ôćĺ `trace-expression`, eval Ôćĺ `eval-expression`).
- In OSS, these env vars are read by the frontend only (chat-with-trace / SQL-with-AI). Signals ÔÇö which also reuses these vars ÔÇö is not an OSS feature; it ships in `lmnr-private`. Don't document Signals as an OSS feature in the OSS README or docker-compose files.
- Provider selection: `LLM_PROVIDER` (=`openai` | `gemini` | `bedrock`) is the single switch. `openai` and `gemini` use `LLM_API_KEY` (+ optional `LLM_BASE_URL`; `openai` is OpenAI-compatible and works with LiteLLM proxy / OpenRouter / vLLM). `bedrock` uses AWS credentials (`AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` / `AWS_REGION`) instead of `LLM_API_KEY`. `LLM_MODEL_SMALL` / `LLM_MODEL_MEDIUM` / `LLM_MODEL_LARGE` are optional overrides; each provider has hard-coded defaults in `DEFAULT_MODELS`. Mirror the app-server's provider matching in `app-server/src/features/mod.rs` / `llm/mod.rs` when extending.
- `Feature.SIGNALS` requires `SIGNALS_ENABLED=true` AND `isAiProviderConfigured()`. `Feature.BATCH_SIGNALS` is currently disabled via the feature flag (`return false`) until the batch path is re-enabled; when re-enabling, gate on `LLM_PROVIDER === "gemini"` since the backend batch API is Gemini-specific.
- The Rust OpenAI client (`app-server/src/llm/openai/`, gated behind the `signals` feature ÔÇö `allow(dead_code)` in OSS) is the path Signals hits. `provider_request_to_openai_body` (`conversions.rs`) must NOT forward `reasoning_effort` when **function tools are present**: gpt-5.x reasoning models 400 on `/v1/chat/completions` with "Function tools with reasoning_effort are not supported ÔÇŽ use /v1/responses instead" ÔÇö this fires on OpenAI direct AND on some OpenAI-compatible gateways (LiteLLM/OpenRouter) that forward to OpenAI, so endpoint-sniffing is unreliable (LAM-1771). The guard is simply `if !has_tools` ÔÇö forward `reasoning_effort` only when no tools are present, regardless of endpoint.
- The `signals` feature does NOT compile standalone in the OSS `lmnr` repo ÔÇö it pulls in clustering worker variants (`BatchWorkerType::ClusteringBatching`, `WorkerType::Clustering`) that live in `lmnr-private`, so `cargo test --features signals` fails with E0599 in `main.rs`. Run `llm/openai` tests under default features (`cargo test --bin app-server llm::openai::conversions`); the test modules are not feature-gated.

## Signals and Alerts

- Alerts reference signals via `alerts.source_id`. There is NO FK constraint from `source_id` to `signals.id` because `source_id` may reference other entity types in the future. When deleting signals, associated alerts must be deleted in application code within the same transaction (see `deleteSignal`/`deleteSignals` in `frontend/lib/actions/signals/index.ts`).
- The Signals sidebar item is behind a feature flag (`Feature.SIGNALS`) which requires `SIGNALS_ENABLED=true` AND a configured AI provider (`LLM_PROVIDER` set to `openai`/`gemini`/`bedrock` with matching credentials). See "AI Features Model Setup" above.
- Alert metadata is stored as JSONB in `alerts.metadata`. For `SIGNAL_EVENT` alerts, it contains `{severity: 0|1|2}` (info/warning/critical). The Rust backend reads this in `postprocess.rs` to filter events by severity threshold, defaulting to CRITICAL (2) when metadata is absent (historical alerts default to the most restrictive level). The frontend edit form also defaults to CRITICAL for alerts without metadata.
- Creating a signal auto-creates a CRITICAL-severity alert and subscribes all workspace member emails as alert targets.
- The `NEW_CLUSTER` alert type and the `skipSimilar` metadata option depend on the clustering service. Gate both the auto-creation of `NEW_CLUSTER` alerts in `createSignal` and UI affordances in `manage-alert-sheet.tsx` behind `isFeatureEnabled(Feature.CLUSTERING)` / `useFeatureFlags()[Feature.CLUSTERING]`. When clustering is disabled, force `skipSimilar: false` on submit so the backend doesn't silently drop notifications.
- In `manage-alert-sheet.tsx`, the `severities` Controller uses `shouldUnregister`. When its render condition (`selectedSignal && alertType === SIGNAL_EVENT`) flips to false ÔÇö e.g. after a successful save resets `selectedSignal` ÔÇö react-hook-form removes the field from form state and `watch("severities")` returns `undefined`. Any `useMemo` that reads `watch`ed values must null-check them (don't call `.length`/`.map` directly) or it will throw synchronously and break the whole page into the error boundary.
- Alert Slack targets persist a channel `id` in `alertTargets.channelId` and the display name in `alertTargets.channelName`. Both columns are populated for every row written by the form, so `resetFormFromSignals` in `manage-alert-sheet.tsx` filters to rows where both are present and uses them directly ÔÇö no name-as-id healing needed.
- `getSlackChannels` in `frontend/lib/actions/slack/index.ts` paginates through Slack `conversations.list` via `response_metadata.next_cursor` (limit=500/page) and returns `{ channels, rateLimited }`. On rate-limit (HTTP 429 OR a 200 OK with `{ok:false, error:"ratelimited"}`) it does NOT retry ÔÇö it returns whatever was collected so far with `rateLimited: true`. The consumer (`manage-alert-sheet.tsx`) toasts when `rateLimited` is true so the user knows the list may be incomplete. We don't cache ÔÇö each request fetches fresh so new/renamed channels appear immediately. The picker (`slack-channel-picker.tsx`) leverages cmdk's `Command` for filtering with a custom `filter` that scores against the channel name (passed via `keywords`); `value` holds the channel id so duplicate names aren't possible.
- The `types` query parameter on `conversations.list` must include both `public_channel,private_channel` ÔÇö dropping `private_channel` silently breaks existing alert targets that point at private channels (they disappear from the picker and can't be re-selected).
- Slack section block `text` fields are hard-capped at 3000 chars ÔÇö Slack rejects the whole `chat.postMessage` call if any block exceeds this. When building alert payloads in `app-server/src/notifications/slack.rs`, feed any user-sourced text (extracted event info, report summaries) through `truncate_to_slack_section_limit` rather than dropping overflowing entries. Count chars, not bytes, and slice on `chars()` to stay on char boundaries (multi-byte payloads will panic on byte-indexed slicing).

## Slack OAuth Broker

- Lets self-hosted instances use Laminar's official Slack app by setting ONE issued key instead of recreating the Slack app + 5 env vars (RFC LAM-1675). The broker runs both OAuth legs and hands the bot token back to the calling instance; message content never transits the broker ("direct-send").
- Instance auth is broker-owned (no license service exists). Each instance gets one key, validated by SHA3-256 hash lookup against the `slack_broker_instances` table (`key_hash` unique). `authenticateInstance` in `lib/actions/slack/broker.ts` returns the `instanceId` or null. Keys are minted with the same primitives as project API keys (`generateRandomKey` + `hashApiKey` in `lib/api-keys.ts`); the hash is one-way (no decoding).
- Three routes under `app/api/broker/slack/`: `/start` (POST, server-to-server) and `/redeem` (POST, server-to-server) both take the instance key in `Authorization: Bearer`; `/cb` (GET) is the ONLY browser-facing leg (Slack's registered redirect target). The bot token never appears in a browser-visible URL ÔÇö `/cb` mints a one-time claim and redirects with only the claim + team name.
- Env vars: `SLACK_BROKER_STATE_SECRET` (HMAC-SHA256 key for the signed `state` blob), `SLACK_CLIENT_ID`/`SLACK_CLIENT_SECRET` (official app creds for `oauth.v2.access`, HTTP Basic auth), and `NEXT_PUBLIC_URL` (derives the registered `redirect_uri` `${NEXT_PUBLIC_URL}/api/broker/slack/cb` ÔÇö must exactly match between authorize and token exchange).
- Security invariants: `state` is HMAC-signed and binds `returnUrl` so a claim can only redirect back to the originating instance; the claim is single-use, stored in Redis cache with a 5-min TTL, removed on redeem, and bound to `instanceId` (`/redeem` rejects claims whose `instanceId` Ôëá authenticated caller).
- The claim ALSO binds `workspaceId` (carried in signed `state`, copied into `ClaimPayload` at `/cb` mint, echoed by `/redeem`). `instanceId` binding alone can't separate workspaces (one instance key for all), and the callback's `workspaceId` rides the public redirect as a mutable query param. Instance-side `redeemBrokeredSlackToken` rejects any claim whose echoed `workspaceId` Ôëá URL `workspaceId` (cross-workspace claim substitution) ÔÇö the broker-bound complement to the callback's `isUserMemberOfWorkspace` check.
- `state` is also single-use: `mintState` records the nonce server-side (`slack_broker_state:<nonce>` Ôćĺ `{instanceId, codeVerifier}`, 10-min TTL) and `/cb` calls `consumeState` (atomic `getAndRemoveIfMatch`, matched by `instanceId`) before code exchange ÔÇö the HMAC alone is replayable. `mintState` is `async`; await it in `/start`.
- PKCE (S256) defends against OAuth token injection (single-use state alone doesn't stop an attacker who leaks a victim's `state`, wins the race to `/cb`, and completes it with their own Slack `code`; a session cookie can't bind the flow since `/start` is server-to-server). `mintState` generates the PKCE pair, stores the secret `verifier` in the server-side nonce record (NEVER in `state`), returns the public `challenge` for the authorize URL; `/cb` reads the verifier via `consumeState` and passes it to `oauth.v2.access`. The direct flow (`connectSlackIntegration`) passes no verifier ÔÇö `exchangeSlackOauthCode`'s `codeVerifier` is optional.
- Single-use state/claim records live in `cache` (`lib/cache.ts`), which falls back to **per-process memory** when `REDIS_URL` is unset ÔÇö a scaled broker without Redis routes `/start`/`/cb`/`/redeem` to different workers and never finds the record. `mintState` calls `assertSharedStore()` to fail loudly at `/start` when `REDIS_URL` is missing. Broker is Cloud-only (Redis always set); the guard catches misconfigured self-hosted deployments.

### Instance-side brokered consumer (self-hosted)
- The instance-side caller lives in the SAME OSS repo and is inert unless `SLACK_BROKER_URL` + `LMNR_LICENSE_KEY` are set. The broker routes and the consumer routes coexist in one codebase: broker routes activate where the official app creds + `slack_broker_instances` rows exist (Cloud); consumer routes activate where the two broker env vars are set (self-hosted). A given deployment is one or the other, decided purely by env. `LMNR_LICENSE_KEY` is the self-hoster's issued license key (sent to the broker as `Authorization: Bearer`) and is intended for reuse by other future paid-feature gates, not Slack-specific.
- Connect flow: `SlackConnectButton` is double-duty ÔÇö **brokered mode takes precedence**: when `brokerEnabled` it links to `/api/integrations/slack/broker-start`; otherwise cloud mode links straight to `slack.com/oauth/v2/authorize` from `SLACK_CLIENT_ID`/`SLACK_REDIRECT_URL`. Broker must win because a broker-configured deployment has no usable Slack app of its own ÔÇö leftover `SLACK_CLIENT_ID`/`SLACK_REDIRECT_URL` env vars must not divert it into the broken direct flow (the button is a client component and can't see `ENVIRONMENT`/secret vars to replicate the server-side `cloudEnabled` gate). On Laminar Cloud the broker vars are never set, so the direct flow still wins there. Wired ONLY through the workspace integration page chain (`page.tsx` Ôćĺ `WorkspaceComponent` Ôćĺ `WorkspaceIntegrations` Ôćĺ `SlackConnectionCard`), NOT onboarding (which builds its own cloud-only URL).
- `/api/integrations/slack/broker-start` (GET) POSTs the broker `/start` with the Bearer instance key (key never enters the browser) and 302s the browser to the returned `authorizeUrl`. The `returnUrl` it sends the broker points back at `/api/integrations/slack` with `workspaceId` (and optional `returnPath`) embedded ÔÇö the broker echoes only claim + team on the way back, so the instance must carry its own workspace context in the returnUrl.
- `/api/integrations/slack` (the existing callback) gained a brokered branch (keyed on the `workspaceId` query param, which the direct path never sets ÔÇö it carries context in `state`) alongside the `code` branch: it calls `redeemBrokeredSlackToken({claim, workspaceId})` (`lib/actions/slack/index.ts`), which POSTs the broker `/redeem` with the Bearer key, then persists via the shared `persistSlackIntegration` (same encrypt + upsert as the direct path). The stored `slack_integrations` row is identical regardless of path, so the Rust send path is unchanged. Both success (with `claim`) and broker-error (`slack=error`, no `claim`) route through this branch so the error lands on the workspace integrations card, not the `/projects` fallback.
- **Brokered `workspaceId` MUST be authorized against the caller's session.** The callback is a public OAuth redirect target and the broker echoes `workspaceId` back only via the URL (it's not bound to the single-use claim). The branch calls `getServerSession` + `isUserMemberOfWorkspace(workspaceId, userId)` before `redeemBrokeredSlackToken` ÔÇö without it, a user who completed their own broker flow could rewrite `workspaceId` to any UUID and store their bot token into a workspace they don't belong to (cross-workspace claim substitution). The redirect runs in the user's browser, so the session cookie is present.
- Crypto: `getSlackKeyFromEnv` (frontend `lib/crypto.ts`) and `decode_slack_token` (`app-server/src/notifications/slack.rs`) both require `SLACK_ENCRYPTION_KEY`. Self-hosters don't set a second key ÔÇö the Helm chart (`lmnr-ai/lmnr-helm`) maps the cluster's `AEAD_SECRET_KEY` secret onto `SLACK_ENCRYPTION_KEY`, so the env var is always populated without an in-code fallback. BOTH sides must stay in sync or encrypt/decrypt diverge.
- `Feature.SLACK` is enabled when EITHER all cloud vars OR `SLACK_BROKER_URL` + `LMNR_LICENSE_KEY` are set.

## Onboarding Wizard

Route groups: `(auth)/layout.tsx` requires a session (redirects to `/sign-in`); the nested `(auth)/(app)/layout.tsx` is the onboarding-cookie gate (redirects to `/onboarding` when the cookie is present). The wizard page is `frontend/app/(auth)/onboarding/page.tsx`. There is NO `(authenticated)` route group ÔÇö the app trees live under `(auth)/(app)/`.

- Wizard entry `frontend/app/(auth)/onboarding/page.tsx`; component `components/onboarding/index.tsx`. Config (`slackClientId`, `slackRedirectUri`, `{workspaceId, projectId}`) flows via `components/onboarding/context.tsx::OnboardingProvider`. Save handlers + `isSubmitting` come from `useOnboardingActions` (`components/onboarding/use-onboarding-actions.ts`); steps consume `useFeatureFlags()` + `useUserContext()` directly.
- Resume state (`{v, userId, workspaceId, projectId, step, startedAt}`) is a 30-day `httpOnly` cookie. **The page (Server Component) only READS it via `getOnboardingState`** ÔÇö Next.js 16 Server Components can't reliably write cookies (silent fail/throw + same-request `redirect()` Ôçĺ redirect loop). All writes go through `app/api/projects/[projectId]/onboarding/state/route.ts` (project-scoped: URL carries projectId, handler derives workspaceId, body is `{ step }`). Membership is enforced by `proxy.ts` gating `/api/projects/*`. Writes happen only once workspace+project exist (after `createWorkspace`, step 0Ôćĺ1); no pre-resource mount-POST. `persistOnboardingStep(projectId, step)` is the only write surface; `finishOnboarding` DELETEs. The handler resets `startedAt` when the cookie's `userId` Ôëá current session, so a stale cross-user cookie is overwritten on first POST.
- The onboarding gate is the `(auth)/(app)/layout.tsx` route-group layout ÔÇö runs once before all app trees. **Do NOT move the gate into `proxy.ts`** ÔÇö that required expanding the matcher to every authenticated prefix, a callback hack to bypass the `/sign-in` short-circuit, and leaking the cookie name into the global pipeline. `proxy.ts` is scoped to API authz (`/api/projects/*`, `/api/workspaces/*`, `/api/shared/traces/*`), the Strapi `/uploads/*` rewrite, and the unauthenticated `/sign-in?callbackUrl=` redirect for app trees ÔÇö it imports nothing from onboarding.
- Resume defaults (`lib/actions/onboarding/resume-defaults.ts::loadOnboardingResumeDefaults`) re-derive form defaults from the DB on every load when workspace/project are bound. Split into per-slice loaders (`loadWorkspaceContext`, `loadSelectedTemplateNames`, `loadSlackConnected`) composed with `Promise.all` ÔÇö keep that shape so queries parallelize. `workspaceName` MUST come from the DB on resume: the plan step passes it into `/checkout` as Stripe `workspaceName` metadata; without it the `"${user.name}'s workspace"` fallback lands in Stripe after a resume.
- The "already has workspaces" legacy redirect to `/projects` fires only when `count > 0 AND no saved cookie owned by THIS user AND slack param undefined`. Narrow `saved` to `ownSaved = saved?.userId === user.id ? saved : null` before every gate ÔÇö an other-user's cookie must look absent, else the wizard renders+overwrites under user B and the gate traps B in a loop until B creates an unwanted workspace.
- Creating a workspace with `isFirstProject: true` (`POST /api/workspaces`) auto-creates the **Failure Detector** signal + trigger + creator email targets inside `createWorkspace` (`frontend/lib/actions/workspaces/index.ts`), same transaction as workspace+project+signal seed ÔÇö so the user is opted into default-report digests by step 1. ## Trace Type Upsert

- The `traces.type` column uses a first-non-zero strategy in the ON CONFLICT upsert: `CASE WHEN traces.type = 0 THEN EXCLUDED.type ELSE traces.type END`. Once a trace is assigned a non-default type (e.g. EVALUATION=1), no subsequent batch can overwrite it. Do NOT revert to `COALESCE` (picks first non-null, lets `0` silently reset a previously set type) or `GREATEST` (picks highest numeric value, which conflates type ordering with type precedence).
- `TraceAggregation::from_spans` in `ch/traces.rs` applies the same rule within a single batch: `trace_type` is only written when `entry.trace_type == 0`, so a child span processed later in the loop cannot overwrite a non-default type already set by an earlier span.

## Signal Triggers

- Signal trigger filters are evaluated in `app-server/src/db/trace.rs` (`matches_filters` / `evaluate_single_filter`). Spans arrive in batches, so filter evaluation must check accumulated state from the DB (e.g. `trace.span_names`) ÔÇö not just the current batch's raw spans. The `traces.span_names` JSONB column aggregates span names across all batches via `||` merge on upsert.
- Trigger evaluation flow: `process_span_messages` Ôćĺ `upsert_trace_statistics_batch` (returns merged DB trace) Ôćĺ `check_and_push_signals` Ôćĺ `matches_filters`. All filters use AND logic.
- Run targeted tests with `cargo test --bin app-server db::trace::tests -- --nocapture`.

## Authentication (Better Auth)

- Auth is **Better Auth** (`better-auth`), not NextAuth. Central config is `frontend/lib/auth.ts` (`auth = betterAuth({...})`); the route handler is `app/api/auth/[...all]/route.ts` via `toNextJsHandler(auth)`. Client helpers live in `frontend/lib/auth-client.ts`; the server-session wrapper is `frontend/lib/auth-session.ts` (`getServerSession()` Ôćĺ `auth.api.getSession({ headers: await headers() })`, shape `{ user: { id, name, email, image } } | null`). Env: `BETTER_AUTH_SECRET` / `BETTER_AUTH_URL`, falling back to `NEXTAUTH_SECRET` / `NEXTAUTH_URL` for compat.
- The drizzle adapter uses `usePlural: true`. Better Auth core appends `"s"` to each model name when `usePlural` is set (`@better-auth/core` `getModelName`), so the jwt plugin's already-plural `jwks` model becomes `jwkss` (no such table Ôćĺ `/api/auth/jwks` 500s). Fix is the jwt plugin's `schema: { jwks: { modelName: "jwk" } }` so pluralization yields `jwks`. Any future plugin whose default model name is already plural needs the same singular-base override.
- Better Auth's adapter writes JS `Date` objects for timestamp columns. The better-auth tables (`sessions`/`accounts`/`verifications`/`jwks`) and the `users.created_at`/`updated_at` columns MUST be declared `mode: "date"` (not `"string"`) in `lib/db/migrations/schema.ts` or user/session creation fails with `ERR_INVALID_ARG_TYPE` (Date passed where string expected). This is a TS-serialization-mode change only ÔÇö the Postgres column type is unchanged, so switching mode needs no new migration.
- JWT parity with the legacy NextAuth JWT session is provided by the `jwt` plugin: `definePayload` emits `{ userId, name, email }`; endpoints `/api/auth/token` (signed EdDSA JWT) and `/api/auth/jwks` (public key set). Workspace-invite tokens are still plain `jsonwebtoken` signed with `BETTER_AUTH_SECRET ?? NEXTAUTH_SECRET` (`lib/actions/workspace/invite.ts`), verified the same way in `app/(auth)/invitations/page.tsx`.
- Client sign-in is unified through `signInWithProvider(provider, callbackURL)` in `auth-client.ts`: github/google use `signIn.social(...)`, while azure/okta/keycloak use `signIn.oauth2({ providerId })` (genericOAuth plugin; provider ids are `microsoft-entra-id`/`okta`/`keycloak`). `signOut()` takes NO `callbackUrl` ÔÇö callers MUST `window.location.href = "/"` (or equivalent) themselves after it resolves.
- `signInWithProvider` MUST pass `errorCallbackURL` (set to `/sign-in?callbackUrl=<original>`) to BOTH `signIn.social` and `signIn.oauth2`. The inline `{ error }` the client returns only covers failures BEFORE the browser redirects to the IdP; once the OAuth round-trip happens server-side, a callback failure (account-linking gate, missing-email rejection) is handled by Better Auth's `redirectOnError` (`oauth2/errors.mjs`), which redirects to the OAuth state's `errorURL`. That `errorURL` comes from the sign-in body's `errorCallbackURL` (`oauth2/state.mjs`) and DEFAULTS to `${baseURL}/error` (i.e. `/api/auth/error`) when unset ÔÇö so without it, IdP failures never land back on `/sign-in?error=<code>` and the sign-in page's `searchParams.get("error")` UI never fires. Better Auth appends `?error=<code>` (separator-aware) itself, so the page reads it directly.
- Passwordless local-email sign-in (legacy Credentials provider parity, self-hosted only) is a custom plugin in `lib/auth-local-email.ts` (`POST /api/auth/sign-in/local-email`), mounted only when `Feature.EMAIL_AUTH`. The pending-invitation processing that the old NextAuth callbacks did now runs in a `databaseHooks` `session.create.after` hook in `lib/auth.ts`. (The legacy NextAuth callback also minted a personal `api_keys` row per user; that table is obsolete/unused ÔÇö see the personal `api_keys` note below ÔÇö so no key is created on sign-up anymore.) The plugin creates users with `emailVerified: true` (NOT `false`): Better Auth's account-linking gate (`requireLocalEmailVerified`, defaults `true` ÔÇö `node_modules/better-auth/dist/oauth2/link-account.mjs`) refuses to link an OAuth identity to a user whose local email is unverified, so a `false` here would brick a later GitHub/Google sign-in for the same address. Local-email is a no-password convenience with no verification state to protect ÔÇö same rationale as the migration backfill of pre-existing IdP users. `EMAIL_AUTH` and OAuth providers are normally mutually exclusive (`features.ts` hides the dummy email input when any provider is configured), so this only bites when `FORCE_EMAIL_AUTH=true` runs both at once.
- **`callbackUrl` is attacker-controlled and must be sanitized to a same-origin relative path** before any client navigation. The local-email flow signs in via a `$fetch` POST (`signInLocalEmail`) that returns JSON and does NOT redirect ÔÇö navigation is a pure client-side `router.push(callbackUrl)` (`components/auth/email-sign-in.tsx`), so Better Auth's `trustedOrigins` (which only guards its own server-side OAuth redirects) never sees it. The trust boundary is the query string entering `app/sign-in/page.tsx` and `app/sign-up/page.tsx`; both call `sanitizeCallbackUrl(searchParams?.callbackUrl)` (`lib/utils.ts`), which rejects anything not starting with a single `/` (absolute `https://evil.com/x`, protocol-relative `//evil.com`, backslash `/\evil.com`, and `\t\n\r`-obfuscated variants browsers normalise) and falls back to `/onboarding`. The OLD `new URL(callbackUrl)` check only rewrote the bare-`/` pathname case and let absolute external URLs flow through ÔÇö an open redirect. The sanitized value also feeds `signInWithProvider`'s `errorCallbackURL`, so OAuth is covered by the same page-level guard.
- `proxy.ts` middleware no longer uses `withAuth`; it calls `auth.api.getSession({ headers: req.headers })` directly. Next.js 16 middleware runs Better Auth on the Node runtime, so the Postgres session lookup works there. The unauthenticated-redirect-to-`/sign-in?callbackUrl=` behaviour for app trees is replicated inline.
- **Migration backfill for existing users** (`0089_lam_1720_better_auth.sql`): adding `users.email_verified boolean DEFAULT false` is NOT enough on its own. Pre-existing users have a `users` row but no `accounts` row (that table is new), so their first OAuth sign-in hits Better Auth's account-*linking* path in `handleOAuthUserInfo`, which is gated on `requireLocalEmailVerified` (defaults to `true`). With every migrated row at `email_verified = false`, that gate returns `"account not linked"` and bricks every existing user. The migration therefore runs `UPDATE users SET email_verified = true` right after adding the column ÔÇö these users were already IdP-authenticated, so they're effectively verified. The `trustedProviders` list does NOT save them (it only exempts the separate `!isTrustedProvider && !userInfo.emailVerified` term). The legacy NextAuth setup was JWT-strategy (zero auth tables, stateless cookie), so the migration to Better Auth's DB-backed sessions also logs every user out once on deploy ÔÇö expected.
- **The personal `api_keys` table is obsolete and unused.** It is NOT the project-level `project_api_keys` table (which app-server auth reads on every request). The user-level `api_keys` table is referenced by **zero** code in either backend (`app-server`, `lmnr-private`) ÔÇö nothing ever reads it as a credential. The legacy NextAuth `createUser` (`lib/db/auth.ts`, now deleted) minted one personal key per user inside the user-creation transaction; Better Auth's auth hooks briefly recreated that, but it has since been removed entirely. No personal key is created on sign-up, and `lib/auth.ts` no longer touches the table. The physical `api_keys` table still exists in the DB (kept for now, not dropped) with its original non-unique `api_keys_user_id_idx` index ÔÇö do NOT re-introduce code that writes to it. If you ever need a per-user credential again, use `project_api_keys` patterns or design a fresh table; don't revive this one.

## CLI authentication (BetterAuth Device Flow Ôćĺ user-scoped tokens)

- The CLI stores **BetterAuth tokens** (user-scoped), NOT a project API key. `lmnr-cli login` walks BetterAuth's native `deviceAuthorization` flow end-to-end (no custom poll endpoint). `lmnr-cli setup` does the same login AND additionally mints a project API key that it writes ONLY to `./.env` (never into `credentials.json`) via our `POST /api/cli/api-key` endpoint.
- **Token model:**
  - **Access token** = a 15m EdDSA JWT minted by `GET /api/auth/token` (with the session token as bearer) Ôćĺ response `{ token }`. Verifiable via `/api/auth/jwks`. This is what app-server will eventually validate statelessly (JWKS). App-server JWT validation is a deliberate LATER phase ÔÇö the CLI's JWT is NOT accepted by app-server yet, so `lmnr-cli sql`/ingestion against app-server with the JWT does not work this iteration.
  - **Refresh token** = the BetterAuth **session token**, returned as `access_token` from `POST /api/auth/device/token`. Durable (session `expiresIn` default 7 days, rolling). Refresh op = re-call `GET /api/auth/token` with the session token Ôćĺ fresh JWT, no browser.
- Endpoints, in `login` flow order:
  - `POST /api/auth/device/code` ÔÇö BetterAuth RFC 8628 code mint. CLI sends `{client_id:"lmnr-cli", scope:"projects:rw"}`.
  - Browser Ôćĺ `/device?user_code=<code>` ÔÇö server component at `app/(auth)/device/page.tsx`. The `(auth)` layout + `proxy.ts` matcher gate unauthenticated visitors to `/sign-in?callbackUrl=/device?user_code=...`. Server-side: claim the code via `auth.api.deviceVerify(...)`, then render **Approve / Deny only** (no project picker ÔÇö the credential is user-scoped, nothing to scope at approval). Approve Ôćĺ `authClient.device.approve({userCode})`; Deny Ôćĺ `authClient.device.deny({userCode})`.
  - `POST /api/auth/device/token` ÔÇö BetterAuth's NATIVE token endpoint. CLI polls here; on approval returns `{ access_token (=session token), expires_in, ... }`. We no longer own a custom poll endpoint.
  - `GET /api/auth/token` (session bearer) Ôćĺ `{ token }` 15m JWT.
  - `GET /api/auth/get-session` (session bearer) Ôćĺ `{ user: { id, email } }` for profile metadata.
- **`POST /api/cli/api-key`** (`app/api/cli/api-key/route.ts`) ÔÇö mints a project API key for the session user (the `setup` flow is the only caller today, but the endpoint is generic, not setup-bound). Auth via session bearer (`auth.api.getSession`). Body (zod `.strict()`): `{ projectId?, deviceName? }`. With `projectId` Ôćĺ membership-check + mint. Without Ôćĺ 1 project mints; >1 returns `400 {error:"project_ambiguous", projects:[{id,name,workspaceName}]}`; 0 returns `400 {error:"no_projects"}`. Returns `{apiKey, apiKeyId, projectId, projectName, workspaceId, workspaceName}`. The minted key goes to `./.env` only.
- **The `bearer()` plugin is required.** `auth.api.getSession({ headers })` only accepts cookie-borne sessions by default; without `bearer()` an `Authorization: Bearer <session-token>` header is ignored. The device-flow `access_token` IS a session token, so the bearer plugin is what lets it round-trip through `getSession()` (used by `/api/auth/token`, `/api/auth/get-session`, and `/api/cli/api-key`).
- **CLI credentials shape.** `~/.config/lmnr/credentials.json` (mode 0600, **keyed by `userId`**, version 1) stores tokens only: `sessionToken` (refresh), `accessToken` (the 15m JWT), `accessTokenExpiresAt` (from JWT `exp`), `sessionExpiresAt?` (from device token `expires_in`), `issuer`, `baseUrl`, `userEmail?`, `userId`, `createdAt`, `lastUsedAt?`. NO project API key, NO `apiKeyId`. `resolveAuth` refreshes the JWT when within ~30s of expiry (re-mint via `/api/auth/token`); a 401 means the session expired Ôćĺ "run `lmnr-cli login`". The returned JWT bearer is not yet accepted by app-server (later phase).
- **Logout** best-effort POSTs `/api/auth/sign-out` (session bearer) to revoke the session server-side, then deletes local creds; `--all` loops all profiles.
- **App-server later phase (two-flow validation):** the eventual design is JWT-via-JWKS for the CLI (user-scoped ÔÇö so app-server must derive the project from the request path / an explicit `--project` arg, since the JWT carries no project) and project API key for the SDK. Not implemented here.
- **`validateClient: async (id) => id === "lmnr-cli"`** in `lib/auth.ts`. Multi-tenant client registration is intentionally out of scope.

## Analytics / PostHog

- Client-side analytics is centralized in `frontend/lib/posthog/`. Feature code should import `track` from `@/lib/posthog` ÔÇö never import `posthog-js` directly.
- Server-side PostHog client lives at `@/lib/posthog/server` (separate import path to avoid bundling `posthog-node` into client bundles).
- The `AnalyticsProvider` (in `lib/posthog/provider.tsx`) handles both PostHog init and user identification. It wraps `PostHogProvider` from `posthog-js/react` so `usePostHog()` still works as an escape hatch.
- Custom events use `track(feature, action, properties?)` which emits `${feature}:${action}`. The `Feature` type is defined in `lib/posthog/client.ts` ÔÇö add new categories there before using them. Use `'sessions'` for trace sessions and `'debugger_sessions'` for CLI debugger sessions ÔÇö these are distinct features.
- All tracking is no-op when PostHog is disabled (`POSTHOG_TELEMETRY !== "true"`). No conditional checks needed in feature code.
- For page-view tracking on server-component pages, use the `PageViewTracker` client component at `components/common/page-view-tracker.tsx` ÔÇö server components cannot call `track` directly (no hooks). For server-action buttons that need tracking, wrap them in a dedicated `"use client"` component that calls `track` before invoking the server action (see `components/invitations/invitation-actions.tsx`).
- `PageViewTracker` deliberately uses an empty `useEffect` dependency array so it fires exactly once on mount. Callers pass inline `properties` object literals (`{ slug }`, `{ traceId }`), which would otherwise create a new reference on every render and re-fire the event.
- Auth flows (sign-in/sign-up) track `*_attempted` before the provider redirect, not on success ÔÇö the OAuth/email flows navigate away before any success callback runs, so the attempt is the last reliable hook. `EmailSignInButton` takes an `action` prop (`sign_in_attempted` | `sign_up_attempted`) because the same component is rendered on both `/sign-in` and `/sign-up`.
- Tracking calls must be guarded by success (`res.ok`). Firing `track(...)` after an unchecked `await res.text()` records events for failed requests and corrupts metrics.
- `track(...)` accepts a 4th arg `{ sendInstantly: true }` that maps to posthog-js's `send_instantly` and bypasses the batching queue. Use it whenever the very next line can tear down the page before the batch flushes: OAuth/email `signIn()` redirects (`components/auth/*`), `router.refresh()` after a mutation (`switch-tier-dialog.tsx`), or any `window.location.assign/href` navigation. Batched events are lost because the request is aborted when the page unloads.
- Server-side captures from Better Auth database hooks (e.g. `auth:user_created` in `lib/auth.ts`) are fire-and-forget: `posthog-node`'s default 10s `flushInterval` schedules a background flush from the active `setTimeout` (which keeps the per-request `PostHog` instance alive until the flush fires), so the Next.js container delivers the event without an explicit `shutdown`/`flush`. Do NOT `await client.shutdown(...)` on the request path ÔÇö it adds up to the shutdown timeout to the login response if PostHog ingest is degraded. Wrap the block in try/catch so analytics failures cannot break login.
- `auth:user_created` fires exactly once per brand-new account, from the `databaseHooks.user.create.after` hook in `lib/auth.ts`.
- PostHog `distinctId` across the codebase is the user's **email**, not the DB user id. The single source of identification is the client-side `identify(email, { email })` call in `lib/posthog/provider.tsx` (mounted from the root `app/layout.tsx` so it runs on every authenticated page). Per PostHog's docs, server-side `posthog.identify(...)` only updates the person profile and CANNOT do the anonymousÔćĺidentified linking the browser SDK does (the server has no anonymous session), so we deliberately don't call `identify` from `posthog-node` ÔÇö it would just add redundant `$identify` events without contributing any new info. Any server-side `client.capture(...)` (e.g. from Better Auth database hooks) must still pass `distinctId: email` directly on each capture ÔÇö passing the DB user id creates a separate PostHog person profile and detaches `$set_once` properties and funnel events from the real user.
- Workspace grouping uses the client-side `WorkspaceGroupTracker` (`components/common/workspace-group-tracker.tsx`), which calls `posthog.group("workspace", workspace.name, { name, id })` on mount. Per PostHog's web-SDK docs, `posthog.group(...)` does two things in one call: emits `$groupidentify` to register/update the group entity's properties AND associates every subsequent session event with `$groups: { workspace: <name> }`. We rely on this so client-side `track(...)` events are auto-tagged for workspace-level analytics ÔÇö do not use server-side `posthog.groupIdentify(...)` instead, since that only updates the group entity and does NOT propagate group context to client events. The tracker is rendered in `app/workspace/[workspaceId]/page.tsx` (inside `WorkspaceMenuProvider`) and `app/project/[projectId]/layout.tsx` (inside `ProjectContextProvider`); render it in a component that already fetches the workspace so we don't add a duplicate query just to access `workspace.name`. Group key is `workspace.name` (readable in the PostHog UI) ÔÇö do NOT key by id, PostHog group filters display the key directly. The tracker's `useEffect` is gated on `[workspaceName, workspaceId]` so it only re-fires on workspace switch, not every render.
- `lib/posthog/client.ts` is a thin wrapper around `posthog-js`: `init()` calls `posthog.init()` once and the other helpers (`identify`/`group`/`reset`/`track`) delegate directly. `posthog-js` handles its own pre-init buffering, so we don't maintain an internal queue or status flag ÔÇö keep it simple.
- `AdvancedSearch` submit tracking lives in a single place: the store's `submit` function (`components/common/advanced-search/store/index.tsx`). `removeTag` and `applyRecentSearch` funnel through `submit`, so they are covered automatically; do not instrument them separately. `addCompleteTag` and `clearAll` do NOT funnel through `submit` ÔÇö we intentionally accept under-counting there in exchange for one consistent metric definition (same `filterCount`/`hasSearch` logic everywhere). The `resource` prop is threaded into the store so the event can label its origin (`traces` / `spans` / `unknown`).
- `AdvancedSearch` autocomplete fetching is gated in `components/common/advanced-search/index.tsx`: SWR only hits `/api/projects/<id>/<resource>/autocomplete` when `resource === "traces" | "spans"` AND no `suggestions` map was passed. Omit the `resource` prop (or pass anything else) on pages that have no autocomplete endpoint (e.g. evaluations) to suppress the request. Tag-level suggestions in `components/common/advanced-search/components/tag.tsx` also silently degrade to `[]` when `AUTOCOMPLETE_FIELDS[resource]` is undefined, so no backend endpoint is required for non-traces/spans consumers.
## Self-hosted Telemetry (anonymous usage heartbeat)

- `frontend/lib/telemetry/` emits an anonymous `self_hosted_heartbeat` PostHog event for Helm/self-hosted deployments. Gated by `Feature.TELEMETRY` (`lib/features/features.ts`): on by default ÔÇö only off when `LAMINAR_TELEMETRY_DISABLED === "true"` (operator opt-out) OR `LAMINAR_CLOUD === "true"` (cloud has first-party analytics). Deliberately does NOT gate on `ENVIRONMENT`, so the many self-hosted installs that don't set `ENVIRONMENT=FULL`/`PRODUCTION` are still observed; the `environment` property on every event lets dev/staging be filtered downstream instead of dropped at the source. Distinct from `Feature.POSTHOG` (that's cloud product analytics keyed on email) ÔÇö telemetry keys on an opaque per-deployment UUID and never carries PII/customer content.
- Identity lives in a **separate `telemetry` Postgres schema** (NOT public), created via boot-time DDL in `lib/telemetry/instance.ts`, NOT a drizzle migration ÔÇö so it can't break installs where the app role lacks CREATE on public, and the OSS migration set stays telemetry-free. The `telemetry.instance` table is a singleton (boolean PK `id` always true) holding `instance_id uuid` + `last_reported_at`.
- Cross-replica dedup is a CAS window-claim: `claimReportingWindow(intervalMs)` does an atomic `UPDATE ... WHERE last_reported_at IS NULL OR last_reported_at < now() - interval`, returning a row only to the winner. The job timer polls hourly but the 6h cadence is enforced by this DB claim, so multiple frontend pods + restarts can't over-report. Wired fire-and-forget from `instrumentation.ts::register()` (outside the `LOCAL_DB` gate, after Laminar init).

## Project Settings (`projects.settings` JSONB)

- Per-project knobs live in `projects.settings JSONB DEFAULT '{}'` ÔÇö one row per project, one column for ALL settings, NOT one column per setting. ### PII redaction (`settings.removePii`)

- Setting `removePii=true` on a project routes every span ingested for that project through the `pii-redactor` gRPC service before storage.
- Frontend toggle: `components/settings/pii-redaction.tsx`, mounted in the General tab. Pro-tier gated in BOTH the UI (greyed `Switch` + "Pro plan required" badge) AND the server action (rejects non-Pro/Enterprise with 403). The server gate is the source of truth; the UI gate is purely UX.
- **The Pro-tier gate only applies on Laminar Cloud.** Self-hosted installs aren't on tiered billing ÔÇö their only seeded `subscription_tiers` row is `unlimited` (`frontend/lib/db/initial-data.json`), which would never satisfy `PRO_LIKE_TIERS`. The two sides must stay in sync (LAM-1731); the server side is authoritative.
- Redaction runs in the consumer (`process_span_messages`) AFTER `build_dedup_batch` and BEFORE the `llm_messages` ClickHouse insert. For dedup'd LLM spans (most LLM spans ÔÇö root spans excepted) `span.input` is `None` on the wire post-LAM-1608, so redaction targets `dedup.messages[k].content` directly: this single buffer is what the `llm_messages` insert reads AND what Quickwit indexing reads via `span_new_message_indices`, so one update covers both storage tiers. Already-seen messages are NOT in `dedup.messages` and ride the wire as hashes only ÔÇö they were redacted on first emit. Root LLM spans (which keep `span.input` populated for the trace-list preview) plus non-LLM spans get whole-`input` redaction. `span.output` is always whole-redacted.

- After redaction mutates `dedup.messages[k].content` in place, the `redact_spans_in_place` write-back loop also patches `dedup.span_content_bytes[dedup_idx]` so the post-dedup input-bytes loop bills the redacted size, matching the "size reflects redacted content" invariant on `estimate_size_bytes`.

## App-server env vars (`src/env/`)

- Every env var the app-server reads is registered in `app-server/src/env/` (one submodule per domain: `server`, `workers`, `batching`, `clickhouse`, `database`, `mq`, `quickwit`, `llm`, `secrets`, `connections`, `rate_limit`, `observability`, `notifications`, `storage`, `sql`, `debugger`, `mock`). - Two kinds of entry:
  - **Typed descriptors** `NumEnv<T>` / `StringEnv` / `BoolEnv` for vars with a static default ÔÇö `pub const FOO: NumEnv<u32> = NumEnv::new("FOO", 4);`, read via `env::workers::FOO.get()`. `.get()` trims, treats empty string as unset, parses, and falls back to the default (empty-string-as-unset matters for k8s ConfigMap keys). `BoolEnv` accepts `true/1/yes/on` + `false/0/no/off` case-insensitively.
  - **Bare-name `&str` constants** (`pub const URL: &str = "CLICKHOUSE_URL";`) for required secrets/connection strings that `expect()` at boot and for vars consumed only inside multi-var feature detection ÔÇö forcing a default onto those would be misleading, so they keep their read logic at the call site and only borrow the name.
  - `env::num_with_default(name, default)` free fn for the rare var whose default is a runtime value (a crate const) and so can't be a `NumEnv` const.
- **`cargo fmt` and `cargo check --features signals` both fail in OSS** because the `signals` feature gates modules (`agent`, `clustering::private`, `signals::private`, `traces::previews`) whose files live only in `lmnr-private`. `cargo fmt` walks the module tree and errors on the missing `mod agent`. To format `main.rs` (or anything that triggers the tree walk), `touch` empty stubs for those four paths, run `rustfmt --edition 2024 <file>`, then delete the stubs. Per-file `rustfmt` on leaf files works without stubs. Default-feature `cargo check`/`cargo fmt` is the real gate.

## Debugger Replay Cache (LAM-1715)

- Server-side cache that lets the SDK replay recorded LLM responses during a debugger re-run. Orchestration lives in `app-server/src/debugger/mod.rs::lookup` (moved out of `traces/debug_cache.rs`); the HTTP surface is `POST /v1/rollouts/{session_id}/cache` (`api/v1/rollouts.rs::lookup_cache`). Cache identity is `(project_id, replay_trace_id, normalize_needle(cache_until))` ÔÇö the `cache_until` window is part of the key so the same trace can be replayed at different cache points within or across sessions; a wider/narrower window is a distinct namespace that warms cold rather than reusing a stale window's entries (and a warm-but-absent hash stays a true terminal MISS *for that window*). `session_id` is in the path for route consistency only and is NEVER part of the cache key ÔÇö warmed content is session-independent, so cross-session sharing of the same trace+window is always safe and is what scoping the lock + ready marker by the needle (not just `(project, trace)`) preserves.
- **Three wire outcomes** (`CacheLookupResponse`, `#[serde(rename_all="camelCase", tag="outcome")]`): `hit` (replay this response), `miss` (warm cache, hash absent Ôćĺ SDK runs live forever), `live` (warmup timed out Ôćĺ run live this call, retry next). MISS Ôëá COLD Ôëá Live: `Live` is ONLY the warmup-timeout path, never an absent hash in a warm cache. COLD never surfaces directly ÔÇö the block-and-warm path turns it into hit/miss/live.
- **Input hash** = `hex(blake3(canonical_json(messages_without_system)))`, computed by `debug_input_hash` in `traces/input_dedup.rs`. This hashes the WHOLE message array (system messages removed) as one blob ÔÇö distinct from the per-message dedup hashing in the same file. **EVERY** `role=="system"` message is stripped regardless of content or position (leading, trailing, repeated) ÔÇö a deliberate `role`-only filter that does NOT reuse `prompt_hash::extract_system_message` (which strips a single system message and only when it has extractable text). The contract is "drop all system messages" because the coding agent edits its own system prompt between iterations; the SDK mirrors this exactly. Keys are sorted recursively, array order preserved, NO number canonicalization in v1. Cross-language parity vectors live at `app-server/test/data/debug/input_hash_vectors.json` ÔÇö the SDKs MUST copy this fixture and produce identical hashes; the app-server impl is the reference.
- **Warmup reads through `spans_v0`**, which is a **parameterized view** (`WHERE project_id = {project_id:UUID}` baked into the view body). Pass project scope via `.param("project_id", project_id)` on the query builder, NOT as a SQL `WHERE` clause ÔÇö see `ch/spans.rs::query_debug_cache_spans_page`. Warmup query selects `LLM` + `CACHED` spans (string `span_type` from the view), `ORDER BY start_time ASC`, paged. Output attributes (`lmnr.sdk.raw.response`, `gen_ai.output.messages`, `gen_ai.response.finish_reason`) are pulled via `JSONExtractRaw(attributes, ...)` (empty string when absent).
- **Tagged response envelope** (DEVIATION the SDK must match): stored `response` is `{ "type": "raw", "response": <lmnr.sdk.raw.response> }` when the raw attribute is present, else `{ "type": "genAi", "messages": <gen_ai.output.messages>, "finishReason": <reason|null> }`. Raw is preferred; a span with neither output source is skipped (its input hash becomes a MISS).
- **`cache_until`** is a span-id needle: hyphen-stripped + lowercased, suffix-matched against `span_id.simple()`, INCLUSIVE of the matched span. Needle genuinely absent (full scan, no match, no ceiling) Ôćĺ empty warm (every lookup a clean MISS, the safe degrade). Ceiling hit before the needle Ôćĺ keep the bounded prefix. A single span whose response alone exceeds `MAX_BYTES` is skipped (like an output-less span) and the scan continues toward the needle ÔÇö only a *cumulative* overflow trips the byte ceiling, so one oversized span can't abort the whole warmup. Dedupe is earliest-by-`start_time` wins, and the earliest occurrence claims its input hash in `seen` *even when its response is uncacheable* (oversized) ÔÇö otherwise a later span with the same input would be cached and a lookup would replay that later call's output instead of degrading to a clean MISS. So an uncacheable-earliest input is a MISS, never a wrong-response HIT. Two size knobs: query page size (strict, 8) vs cache ceiling (`MAX_SPANS` 256 / `MAX_BYTES` 64 MiB).
- **Spawn-and-wait warmup**: one winner takes a Redis lock (`try_acquire_lock`), `tokio::spawn`s `warm` (which writes entry keys + a ready marker, all with TTL ÔÇö even an empty warm sets the marker so COLD can't persist), then releases the lock. Winner AND waiters both poll the ready marker up to `WARMUP_TIMEOUT_SECONDS`; on timeout they return `Live`. The warm runs a sibling heartbeat task that `renew_lock`s the lock every ~`LOCK_TTL_SECONDS/3` so a warm that paginates a large trace can outlive a single `LOCK_TTL_SECONDS` without the lock expiring and a second lookup double-warming; the heartbeat is `abort()`ed before the lock is released. `renew_lock` is a `CacheTrait` method (Redis `EXPIRE`, returns false when the key is already gone; in-memory updates the `locks` map and reports false for an expired entry) ÔÇö note `set_ttl` does NOT touch the in-memory `locks` map, which is why a dedicated renew primitive exists. Six env knobs, all `DEBUGGER_CACHE_*` (`QUERY_PAGE_SIZE`, `MAX_SPANS`, `MAX_BYTES`, `TTL_SECONDS`, `LOCK_TTL_SECONDS`, `WARMUP_TIMEOUT_SECONDS`), read once via `LazyLock`. Key-name constants are in `cache/keys.rs` (`DEBUGGER_CACHE_KEY` / `_READY_KEY` / `_LOCK_KEY`); concrete keys are `{KEY}:{project_id}:{trace_id}:{needle}[:{input_hash}]` where `needle = normalize_needle(cache_until)` ÔÇö all three keys (entry, ready, lock) carry the needle, so lock granularity matches content granularity (same trace+window Ôćĺ one shared warmup; different windows Ôćĺ independent, non-contending warmups).
- The pure selection logic (`select_entries`) takes explicit `max_spans` / `max_bytes` params so dedupe / ceiling / needle-boundary behaviour is unit-testable without a live ClickHouse. Edition-2024 let-chains (`if ÔÇŽ && let Ok(x) = ÔÇŽ`) are used throughout ÔÇö this crate is on Rust edition 2024.

## Project-scoped shared content dedup (`shared_content` / `spans_v0`)

- **`shared_content` table** (`46_shared_content_project_scope.sql`) is `(project_id, content_hash) Ôćĺ content` ÔÇö project-scoped, not trace-scoped. Same content shared across traces collapses to one row. Backs three paths: input messages, output messages, and tool-definition blobs (hash space is shared; the table doesn't know which path produced a row).
- **Forward-only migration**: `llm_messages` (trace-scoped) is NOT dropped; `spans_v0` reads with `coalesce(dictGetOrNull('shared_content_dict', ...), dictGetOrNull('llm_messages_dict', ...))` so legacy spans keep resolving via the trace-scoped dict. New writes only go to `shared_content`. Self-hosted operators upgrade in a single release; backfill is optional / future work. Dropping `llm_messages` is a separate later migration.
- **`shared_content_dict`** is created at frontend boot (`ensureSharedContentDict` in `frontend/instrumentation.ts`), same `CREATE OR REPLACE` env-driven pattern as `llm_messages_dict` and for the same reason (clickhouse-migrations checksum guard rules out baking creds into a migration). Recreating both on boot is cheap (DDL only); both are `COMPLEX_KEY_CACHE(SIZE_IN_CELLS 131072)` with `LIFETIME(MIN 30 MAX 60)`.
- **Two Redis scopes, two semantics**: `s:{project}:{hash}` is project-scoped storage presence (drives wire content + `shared_content` insert). `tn:{project}:{trace}:{hash}` is trace-scoped first-occurrence (drives `*_new_message_indices` for search). They are independent: a message can be storage-hit + trace-miss (don't ship content, but DO mark trace-new) ÔÇö that's the cross-trace case that keeps span search returning the right span per trace under project-scoped storage.
- **Stamping invariant**: producer never writes Redis. `mark_seen` is the sole writer and runs on the consumer **only after BOTH the `shared_content` AND the `spans` inserts succeed** (last step of the span branch). The two key axes have different backing tables: `s:` (storage) keys are backed by `shared_content`, but `tn:` (trace-new) keys are backed by `spans.*_new_message_indices`. Stamping `tn:` before the spans insert (the old `shared_content Ôćĺ mark_seen Ôćĺ spans` order) left a window where a permanently-dropped spans insert orphaned the trace-new marker ÔÇö later spans in the same trace saw the `tn:` key and shipped empty `*_new_message_indices`, so no span recorded the first occurrence. Insert failure Ôćĺ return transient Ôćĺ Rabbit redelivers Ôćĺ no phantom keys for data that didn't land in CH.
- **Output dedup uses the same `shared_content` table**. CHSpan gets `output_message_hashes Array(FixedString(32))` + `output_new_message_indices Array(UInt16)`. The `seen_storage_in_batch` HashSet is shared across input + output + tools so a hash that appears as input in span A, output in span B, and inside a tool blob in span C emits exactly one row, billed to the first referrer in batch order.
- **Tool definitions** are normalized into a single canonical JSON array per span by `traces/tool_dedup.rs::extract_tool_definitions` from `ai.prompt.tools` (Vercel AI SDK), `gen_ai.tool.definitions` (OTel GenAI), or `llm.request.functions.{N}.*` (OpenLLMetry/LangChain). The producer hashes the array as one blob and stores it on `CHSpan.tool_definitions_hash FixedString(32)`. Empty/zero hash = no tools. Attributes-tab does NOT show tool definitions anymore; `spans_v0` exposes a virtual `tool_definitions` column reconstructed via `dictGet('shared_content_dict', 'content', (project_id, tool_definitions_hash))`.
- **`should_keep_attribute`** strips `ai.prompt.tools`, `gen_ai.tool.definitions`, and any key starting with `llm.request.functions.` so that legacy spans reaching the consumer without a producer-built `tool_dedup` verdict don't surface tool defs in `CHSpan.attributes` (avoids double-counting bytes AND prevents stale tool data in the attributes tab). Producer also strips them at extraction time so they don't ride the wire.
- **Frontend**: `Span.tools?: string` from `spans_v0`. `extractToolsFromColumn(tools)` is the primary path; `extractToolsFromAttributes(attrs)` is the legacy fallback. `resolveTools(span)` (in `tool-list.tsx`) prefers the column and falls back to attribute extraction; call sites use `resolveTools(span)`.
- **Wire shape**: `RabbitMqSpanMessage` carries three `Option<...>` fields with `serde(default)`: `input_dedup: Option<MessageDedup>`, `output_dedup: Option<MessageDedup>`, `tool_dedup: Option<ToolDedup>`. Legacy producers (no dedup fields) still work ÔÇö the consumer's per-span resolution treats `None` as "no dedup applies", so `CHSpan` falls through to raw `input` / `output` / no `tool_definitions_hash`.
- **Search snippets** (`search/snippets.rs::build_snippet_query`) tries `shared_content_dict` first then falls back to `llm_messages_dict` for input. Output snippets read only `shared_content_dict` (no legacy data exists). Both still scope to `*_new_message_indices` so search semantics are "first occurrence per trace" ÔÇö preserved across project-scoped storage exactly because trace-new is its own Redis scope.
- **Billing accounting** in `process_span_messages`: input bytes follow the existing model (32B/hash + first-referrer content_bytes for dedup'd, raw JSON otherwise). Output bytes added on top ÔÇö only counted when there's an output dedup verdict; non-dedup output is already in `estimate_size_bytes` via `span.output`. Tool bytes: 32B for the hash + first-referrer content if any; `should_keep_attribute` removed the source attribute bytes from `attributes` accounting so this isn't double-counted.

## LLM span input deduplication (`llm_messages` / `spans_v0`)

- LLM span `input` is structurally deduplicated: each message in the JSON array is BLAKE3-256 hashed over **sorted-key canonical JSON** (so field-order-only diffs collapse), emitted as a row in `llm_messages` (ReplacingMergeTree, `ORDER BY (project_id, trace_id, message_hash)`, **trace-scoped** ÔÇö same content in two traces = two rows), and referenced via `spans.input_message_hashes Array(FixedString(32))` while `spans.input` is stored empty. Stored `content` is the message's **original-order** `serde_json::to_string` run through `sanitize_string` (serde built with `preserve_order`), so reconstruction is byte-identical to the non-dedup path; the hash is over canonical JSON for stable dedup identity. Non-LLM / tool-span `input` is never touched.
- `spans_v0` reconstructs via `arrayMap(h -> dictGetOrDefault('llm_messages_dict', 'content', (project_id, trace_id, h), 'null'), input_message_hashes)`, falling through to raw `input` when `input_message_hashes` is empty. The `llm_messages_dict` is a COMPLEX_KEY_CACHE dictionary **created at frontend boot** (`ensureLlmMessagesDict` in `frontend/instrumentation.ts`, `CREATE OR REPLACE` with `CLICKHOUSE_USER/PASSWORD/DB` interpolated into `SOURCE`), NOT by migration ÔÇö `clickhouse-migrations` MD5-checksums `.sql` files and would break on credential rotation, so migration 43's credential-less version is dropped by migration 45 (`45_drop_llm_messages_dict.sql`). Boot recreate is cheap (lazy cache, `LIFETIME(MIN 30 MAX 60)` refreshes within ~1min) and runs before Next.js serves traffic. `dictGetOrDefault` keyed on `(project_id, trace_id, message_hash)` avoids the earlier LEFT JOIN + `groupArray` shape (CH can't push the outer `trace_id` predicate through `GROUP BY`, so it aggregated all project messages). Dict attrs can't be `FixedString(N)` on CH 25.12 (`UNKNOWN_TYPE`) ÔÇö `message_hash` is `String` and CH coerces the `FixedString(32)` transparently.
- `spans_v0` exposes `multiIf(...) AS span_type` which collides with `spans.span_type UInt8` under CH 25.x's analyzer. Migration 43 sidesteps without a rename via `ALTER TABLE spans ADD COLUMN span_kind UInt8 DEFAULT span_type`; the view reads `span_kind`, ingestion keeps writing `span_type` (`CHSpan.span_type` in `app-server/src/ch/spans.rs`). Do NOT drop `span_type` ÔÇö ingestion depends on it; `span_kind` is the DEFAULT-derived shadow column.
- **Billing / size attribution:** `span.estimate_size_bytes()` does NOT count `input`; the post-dedup input-bytes loop in `process_span_messages` (`app-server/src/traces/processor.rs`) owns it for every span (runs before both `CHSpan::from_db_span` and `update_workspace_bytes_ingested`, covers non-recorded spans too). Branches: dedup'd LLM spans pay `32 * num_hashes + span_content_bytes` (`span_content_bytes` = newly-inserted `llm_messages.content` only, so a shared message is billed once to the first referrer; `build_dedup_batch` returns it aligned with `span_hashes`); non-recordable LLM spans (e.g. CC Bash `anthropic.messages` via `is_skip_cc_span`, excluded from `recordable_indices` so no `dedup_lookup` entry) are charged from their own `dedups[span_idx]` (`hashes.len()*32 + sum(new_contents lens)`) ÔÇö a na├»ve `estimate_json_size(span.input)` fall-through would charge 0; everything else pays `estimate_json_size(span.input)`. `estimate_size_bytes` filters `raw_attributes` via `should_keep_attribute` (matching `CHSpan.attributes`) because parsers COPY input-bearing attributes into `span.input` without removing them from `raw_attributes` ÔÇö without the filter every OTel LLM/tool span double-bills. The loop MUST run AFTER `convert_span_to_provider_format` (LangChain conversion rewrites `span.input` to a differently-sized `Value::Array`).
- CH 25.x views do NOT support correlated scalar subqueries (both `SELECT (SELECT content FROM llm_messages WHERE message_hash = h)` and `WITH (SELECT ... WHERE trace_id = s.trace_id)` fail `NOT_IMPLEMENTED: can't find correlated column` at SELECT time though `CREATE VIEW` succeeds) ÔÇö use the dictionary lookup.
- Redis key `m:{project_id_simple}:{trace_id_simple}:{hex(hash)}` TTL 1h, hot-path "seen recently" filter. MUST include `trace_id` (table is trace-scoped; a project-scoped key would suppress an insert for a message seen in another trace, leaving dangling `input_message_hashes`). Best-effort: on `exists()` error always insert, RMT merges. `mark_seen` is the only Redis writer, runs AFTER a successful `llm_messages` insert; never stamp before insert (a transient failure would leak `m:` keys whose rows don't exist, surfacing as literal `null` in reconstructed input).
- Ingest order in `process_span_messages` (LAM-1609): build dedup batch synchronously, then `tokio::join!` two branches ÔÇö (a) trace upsert (PG `traces` Ôćĺ CH `traces` Ôćĺ realtime) and (b) the **sequential** span path `shared_content Ôćĺ spans Ôćĺ mark_seen`. Span path stays sequential because `spans` is plain MergeTree (NOT RMT): inserting `spans` before `shared_content` and having `shared_content` fail would double-store every span on retry. `mark_seen` is LAST so `tn:` keys aren't stamped until backing rows are durable. `build_dedup_batch` dedupes `(project_id, trace_id, hash)` triples via HashSet (key MUST match table ORDER BY; a batch can mix projects). Do NOT collapse into a 3-way join.

### Producer-side dedup (LAM-1608)

- Hashing + Redis-existence checks ALSO run on the producer (HTTP / gRPC ingest in `publish_span_messages`) BEFORE the message hits Rabbit. Already-seen LLM input messages are stripped from `span.input`; the wire payload carries an `LlmInputDedup` (`hashes`, `new_indices`, `new_contents`) instead. Wire savings grow linearly with conversation history depth, which is the point.
- `RabbitMqSpanMessage` carries two new fields with serde defaults: `pre_processed: bool` (consumer skips `parse_and_enrich_attributes` + `convert_span_to_provider_format` when true) and `input_dedup: Option<LlmInputDedup>` (the producer's hash + Redis-status verdict).
- Consumer ordering invariant: `prepare_span_for_recording` runs unconditionally (handles usage / span path / status), `convert_span_to_provider_format` runs ONLY when `pre_processed=false` ÔÇö running it twice would double-apply the LangChain rewrite (which replaces `span.input` with a fresh `Value::Array`) and the producer's hashes would no longer match the consumer's view of the input.
- `span.input = None` on the wire when a plan was produced ÔÇö EXCEPT for spans the consumer would render as the trace's root, because `TraceAggregation::from_spans` reads it for `root_span_input` (truncated preview rendered in the trace list). The producer carve-out matches the consumer's promotion logic, not just OTel parentage: keep `input` if `parent_span_id.is_none()` (natural OTel root) OR `is_top_span(span, &span.attributes)` (Laminar SDK top span ÔÇö `span_id == ids_path[0]` AND `name == path[0]`; `prepare_span_for_recording` will later null its `parent_span_id`). The na├»ve `parent_span_id.is_some()` check would let an OTel-parented top span ride the wire with `input = None` and then get promoted to root on the consumer, silently erasing `root_span_input`. `is_top_span` is `pub(crate)` in `traces::utils` for this reason; do not duplicate the heuristic. Root spans are 1 per trace; dedup savings come from the long tail of nested LLM spans either way.
- **Producer is the only dedup site.** `preprocess_for_queue` in `producer.rs` is the one place that runs `build_dedup`; the consumer never re-mints. A `None`-dedup span reaching `build_dedup_batch` is treated as "no dedup applies": empty hashes, `CHSpan.input` keeps the raw JSON written by `from_db_span`, and `spans_v0` falls through to raw `input` via its `if(notEmpty(input_message_hashes), <reconstruct>, input)` branch. This means any ingest path that bypasses `publish_span_messages` (currently only `routes/spans.rs::create_span`, which always writes `input: None` anyway) is stored the old way without dedup ÔÇö correct, just uncompressed. Adding a new bypass path that ships populated array input would silently miss dedup compression; if that becomes a concern, gate it with an `assert!` / `log::warn!` at the consumer rather than re-introducing the mint loop.
- `build_dedup_batch` is hardened against malformed dedups: `dedup.hashes.get(pos as usize)` and `dedup.new_contents.get(i)` are bounds-checked with `else continue`, so a corrupt-on-deserialise dedup or a future producer bug skips the bad entries instead of panicking and looping the entire flush on requeue.
- Quickwit indexing reads pre-built `dedup.span_new_message_values` (a `Vec<Vec<Value>>` aligned with the spans batch) instead of projecting indices back onto `span.input` ÔÇö required because `span.input` is `None` on the producer path.
- Producer NEVER writes Redis. `mark_seen` is the only writer, runs only on the consumer, AFTER a successful `llm_messages` insert. The "stamp only after success" invariant is preserved exactly ÔÇö the producer's Redis check is read-only and best-effort (errors fall through to "treat as new", same as the consumer's prior behaviour).
- `preprocess_for_queue` in `app-server/src/traces/producer.rs` runs sequentially per-span (NOT parallel via rayon) ÔÇö Redis `exists()` calls are cheap and we don't want to thunder-herd Redis on large batches. Most ingest calls carry 1-N spans; per-span async loop is the right granularity.

## Quickwit indexing of LLM spans (LAM-1599)

- Quickwit only indexes the **new** subset of an LLM span's input ÔÇö the messages this span was first to introduce in its trace. Older repeated history is searchable via the earlier span that introduced it. Implementation: `build_dedup_batch` in `app-server/src/traces/input_dedup.rs` returns `span_new_indices: Vec<Vec<u16>>` (0-based positions inside `span_hashes[i]`); `process_span_messages` projects those indices back onto `span.input` (`build_new_messages_subset`) and `QuickwitIndexedSpan::from_span` indexes the resulting array. Non-LLM spans / non-array input fall through to raw `span.input`.
- The same indices land in ClickHouse as `spans.input_new_message_indices Array(UInt16) CODEC(ZSTD(3))` (migration `44_spans_input_new_message_indices.sql`). `CHSpan` only populates them when `input_message_hashes` is non-empty; legacy rows have an empty array, which the search snippet query handles gracefully (empty `arrayMap` Ôćĺ empty input snippet, output / attributes still match).
- Search snippets read directly from the raw `spans` table (NOT `spans_v0`): for spans with `notEmpty(input_message_hashes)` the input snippet is built by `arrayMap(i -> dictGetOrDefault('llm_messages_dict', 'content', (project_id, trace_id, input_message_hashes[i + 1]), 'null'), input_new_message_indices)` joined with `,`; non-LLM spans (empty `input_message_hashes`) fall back to `extract(input, ...)` so their raw `spans.input` is still searchable. Output / attributes use `extract(...)` unconditionally. Single CH query, no parallel split. CH arrays are 1-indexed, hence `[i + 1]`. See `build_snippet_query` in `app-server/src/search/snippets.rs`.
- Cleaning runs at `from_span` build time, not in the consumer. `clean_for_indexing(s, strip_roles)` in `app-server/src/quickwit/preprocess.rs` chains `ANSI strip Ôćĺ strip_noise (base64 images, signature / thought_signature values) Ôćĺ optional strip_role_keys Ôćĺ clean_whitespace (collapse real + literal `\n\t\r`, drop other backslashes) Ôćĺ unicode whitespace classes Ôćĺ NFC`. LLM spans pass `strip_roles=true` for input/output; attributes always pass `false`. The chain only removes / collapses chars (never adds), so the snippet `extract()` regex with `[\s\S]{0,50}` context and `[^a-zA-Z0-9]+` token gaps still matches the reconstructed-and-cleaned text.
- Events keep their own `preprocess_text` path (`PreprocessForIndexing` impl on `QuickwitIndexedEvent`); spans' `PreprocessForIndexing` is a no-op because cleaning already happened upstream.

## Quickwit indexing of signal events (LAM-1652)

- Signal events get their own Quickwit index `signal_events` (`frontend/lib/quickwit/indexes/signal_events.yaml`, applied at boot by `initializeQuickwitIndexes`). `mode: dynamic` (like `spans_v2.yaml`); the free-text content field is `payload`, declared `type: json`. `default_search_fields` is `[]` because the json field type **rejects bare root queries** ÔÇö every search addresses subfields (`payload.foo:"ÔÇŽ"`), built by the search route. Structural ids (`project_id`, `signal_id`, `trace_id`, `id`) are tokenizer:raw + fast; `severity` is `u64` (Quickwit rejects `u8`).
- **`payload` ships as a raw JSON object**, not flattened/filtered: `QuickwitIndexedSignalEvent.payload` is `serde_json::from_str(&event.payload).unwrap_or(Value::Null)`. The json field type replaces the old flatten-with-allow-list because: (1) **per-subfield position streams** ÔÇö each key gets its own positions so phrase queries (all user queries are phrases via `escape_quickwit_query`) can't span field boundaries (the old space-join produced cross-field adjacency false positives); (2) **type inference** ÔÇö numeric/boolean subfields indexed under inferred types, so a string-path `OR` can't match a numeric column; (3) **keys are field paths, not tokens** ÔÇö searching `"prompt"` doesn't false-match every event with a `prompt` column.
- Producer hook `publish_signal_events_for_indexing(events, queue)` in `app-server/src/quickwit/producer.rs` mirrors the span/event indexer (`IndexerQueuePayload::SignalEvents` in `quickwit/mod.rs`, routed by `QuickwitIndexerHandler` in `consumer.rs`). Called in the private signals handler (`app-server/src/signals/private/response_processor.rs`, after `insert_signal_events` succeeds, before clustering). Errors logged + swallowed (a Quickwit hiccup never blocks signal-event creation).
- `flatten_json` is a no-op (ships `Value` as-is for per-subfield indexing); `preprocess_for_indexing` walks the `Value` and applies `preprocess_text` (ANSI strip + whitespace collapse + NFC) to every string LEAF via `preprocess_json_strings` (`quickwit/utils.rs`). Leaf-aware because collapsing the whole JSON to a string would break per-subfield indexing.
- **`payloadFields` is load-bearing for search, not just snippets.** Used for (1) building the Quickwit query `payload.f1:"ÔÇŽ" OR payload.f2:"ÔÇŽ"` (json field has no searchable root) and (2) per-field CH `extract()` for snippet offsets. Frontend filters to `f.type === "string"` before sending. `search_signal_events` short-circuits to empty hits when `payload_fields.is_empty()` (a query with no clauses is rejected by Quickwit) ÔÇö there is NO all-fields fallback; new callers MUST send a non-empty list.
- **Field-name injection is blocked at query construction:** `search_signal_events` filters every `payload_fields` entry against `PAYLOAD_FIELD_NAME_RE` (`^[a-zA-Z_][a-zA-Z0-9_]*$`) before interpolating into `payload.{name}:{escaped_query}`. The form rule in `schema-field-row.tsx` is the only prior validation (server zod is `z.record(z.string(), z.unknown())`), so without the Rust filter a name like `f1:"x") OR (payload.f1` would escape the parenthesised tenant scope and search the whole index cross-tenant. Failing names are silently dropped (unsearchable but still render). Mirror this filter anywhere user-derived field names hit Quickwit syntax.
- `CHSignalEvent.timestamp` is `i64` nanos-since-epoch (CH `DateTime64(9,'UTC')`); `from_event` reconstructs via the infallible `DateTime::<Utc>::from_timestamp_nanos`, and Quickwit's `timestamp` field accepts the RFC3339 serialisation natively.
- Search route `POST /api/v1/projects/{projectId}/signal-events/search` (`app-server/src/routes/signal_events.rs`) wraps `crate::search::signal_events::search_signal_events`. Returns `{ hits: SignalEventSearchHit[] }`, each `{ id, fieldSnippets: Record<fieldName, SnippetInfo> }`. With `payloadFields: string[]` it runs a CH `extract()` per field over the real `signal_events.payload` for highlight offsets; empty `payloadFields` returns ids only. No `snippetsCount`/`+N` badge (every matching field renders in its own column).
- Snippet fetch `fetch_signal_event_snippets` (`app-server/src/search/snippets.rs`): `SELECT id, array(extract(coalesce(nullIf(JSONExtractString(payload,'<f>'),''), toString(JSONExtractRaw(payload,'<f>'))), '<regex>'), ÔÇŽ) AS field_snippets ... WHERE project_id=? AND signal_id=? AND id IN (ÔÇŽ)`. Positional `Array(String)` zipped against the field-name list. `JSONExtractString` unescapes so offsets land on glyphs; `JSONExtractRaw` fallback covers non-string fields. `signal_events` is plain MergeTree scoped by `(project_id, signal_id)` ÔÇö no FINAL. `build_search_regexes` / `post_process_snippet` / `escape_quickwit_query` (pub(crate) re-export from `search/mod.rs`) are shared with the spans path ÔÇö don't fork. Quickwit query: `project_id:{pid} AND signal_id:{sid} AND (payload.f1:"ÔÇŽ" OR ÔÇŽ)`; `search_field` not set (each clause carries its path).
- **Filter vs highlight can disagree on tokenisation edge cases** (unicode whitespace, hyphenation ÔÇö Quickwit `default` tokenizer Ôëá CH regex byte-for-byte) Ôćĺ a row with no highlighted cell. Cross-field adjacency false positives are structurally impossible now. Do NOT re-filter hits against snippet emptiness ÔÇö it drops valid matches.
- Time bounds attached to the Quickwit body only when supplied; the events CH query is unbounded with no params, so a default window in search would drop older matches. Backfill NOT included ÔÇö pre-existing events show in the table but aren't searchable until re-indexed.
- **Frontend flow:** `searchSignalEvents` (`frontend/lib/actions/events/search.ts`) returns `SignalEventSearchHit[]`; `getEventsPaginated` / `getEventsByEmergingClusterPaginated` build `idFilter` from hit ids, retain the list, then `attachSnippets` merges `fieldSnippets` onto each `EventRow` by id. The events route accepts repeated `payloadField=<name>` params (only when `search` active). `createPayloadColumnDef` (`frontend/components/signal/events-table/columns.tsx`) renders `<HighlightedSnippet>` (UTF-16 `<mark>` slices) when `fieldSnippets?.[field.name]` is set, else the normal cell. Table UI uses `AdvancedSearch` (`events-table/index.tsx`); `resource` widened to include `"signal-events"` (`advanced-search/types.ts`), which disables autocomplete (no endpoint; `autocompleteResource` gate restricts to `"traces"|"spans"`). Recent searches persist via Zustand keyed `signal-events-${signal.id}`.

## Post-factum Trace Metadata (`POST /v1/traces/metadata`)

- Customers patch trace metadata after ingestion via `POST /v1/traces/metadata` `{ traceId, metadata }`. The handler (`api/v1/traces_metadata.rs`) verifies the trace exists in the project (returns 404 otherwise) and pushes a virtual span carrying the metadata as `lmnr.association.properties.metadata.<key>` attributes plus the `lmnr.internal.metadata_only` marker (`SPAN_METADATA_ONLY`). Reusing the regular `observations_queue` (via `publish_span_messages`) means concurrent metadata patches and span ingestion for the same trace serialise on Postgres' implicit row lock for `(project_id, id)` ÔÇö no separate locking required.
- Consumer split (`process_span_messages`): metadata-only spans are pulled out of `messages` BEFORE the regular pipeline (after `parse_and_enrich_attributes`, before usage lookup), converted to `TraceMetadataPatch { trace_id, project_id, metadata }`, and applied via `merge_trace_metadata_batch` (`db/trace.rs`) ÔÇö `UPDATE traces SET metadata = COALESCE(traces.metadata || $patch, ...), num_spans = COALESCE(num_spans, 0) + 1` against existing rows only. Patches that target a non-existent trace are silently skipped (no stub row created); the route's existence pre-check covers the normal case, and this guard handles the rare "trace deleted between request and consumption" race. The patch UPDATE runs AFTER `upsert_trace_statistics_batch` in the same `trace_branch`.
- **The patch UPDATE bumps `num_spans` by 1.** `traces_replacing` is `ReplacingMergeTree(num_spans)` and we MUST give the patched row a strictly higher version than the row written by the same-batch aggregation upsert ÔÇö otherwise CH's merge winner is undefined and the pre-patch metadata could survive. The +1 is paid by the virtual metadata-only span that drove the patch, so `num_spans` stays consistent with "one virtual span ingested" (it's still NOT charged in `spans` / Quickwit / billing ÔÇö only the trace-level counter moves). End-state `num_spans = real_spans + count(metadata patches)`. **Other trace stats are untouched**: `end_time`, `start_time`, tokens, cost, top_span, tags, span_names, status do NOT change.
- `updated_traces` is deduped by `(project_id, id)` before the CH insert, keeping the latest occurrence. With the `num_spans` bump above, this is now an **optimization** (RMT would pick the patched row anyway because its version is higher) rather than a correctness requirement ÔÇö the dedup avoids a redundant insert / extra part on the hot ingest table. Metadata patches always extend `updated_traces` after the aggregation results, so last-write-wins keeps the patched row.
- The two PG steps' failure modes are **independent**: a single flush can mix span ingestion for trace A with a metadata patch for unrelated trace B, so `upsert_trace_statistics_batch` errors must not drop B's patch. `trace_branch` collects whatever succeeds and dispatches it to CH / realtime.
- Signals (`check_and_push_signals`) are gated on the aggregation upsert specifically: `trace_branch` returns `Some(aggregation_traces)` ONLY when the aggregation upsert ran AND succeeded. A pure metadata-only flush (no real spans, only `POST /v1/traces/metadata` virtual spans) skips signal evaluation entirely. Even on a mixed flush, only the aggregation traces ÔÇö not the patched-only traces ÔÇö are passed to the signals call. Metadata patches don't touch any field signals evaluate, and feeding a patched-only trace through `matches_filters` would re-fire any signal that already triggered.
- `should_record_to_clickhouse` returns `false` for metadata-only spans, but the actual filter happens earlier (the `messages.retain` in the processor) so they never reach `recordable_indices`, `CHSpan` build, dedup, Quickwit indexing, autocomplete, realtime span updates, or workspace-bytes billing. They contribute nothing to `TraceAggregation` either ÔÇö only the explicit `num_spans = num_spans + 1` in the patch UPDATE moves the count.

## Span-view Message Parsing

- Span input/output JSON is normalized into one of several provider shapes by `processMessages` in `frontend/components/traces/span-view/messages.tsx`. Detection order matters: signals-gated Anthropic Ôćĺ OpenAI Chat Completions Ôćĺ OpenAI Responses Ôćĺ LangChain Ôćĺ Anthropic fallback Ôćĺ Gemini Ôćĺ generic. Only message items carry a `role`. For role-colored headers in the Messages virtualizer, `responsesItemRole` in `messages.tsx` synthesizes a role: tool-call items Ôćĺ `assistant`, tool-output items Ôćĺ `tool`, `reasoning` Ôćĺ `assistant`.

When OpenAI adds a new field, add it to the schema rather than reintroducing `.loose()`.
- When adding a new provider format, update `ProcessedMessages`, `processMessages`, `buildToolNameMap`, and `renderMessageContent` in `messages.tsx`, and add a renderer component. Tool-call IDs are mapped to tool names via `buildToolNameMap` so tool-result items can show their originating tool name even when the output item only carries `call_id`. ## ClickHouse async-insert tuning

- The global CH client in `app-server/src/main.rs` sets `async_insert=1, wait_for_async_insert=1` on every insert. With CH's adaptive busy-timeout (`async_insert_use_adaptive_busy_timeout=1`), the server raises the per-buffer wait toward `async_insert_busy_timeout_max_ms` (default 1000 ms) whenever flushes are time-triggered rather than size-triggered. Our consumer-side batchers already coalesce rows upstream, so CH's per-flush byte size is far below `async_insert_max_data_size` and every flush is time-triggered ÔÇö the adaptive loop parks at `max_ms` and `wait_for_async_insert=1` makes the client block for that full duration (LAM-1628).
- The hot ingest tables (`spans`, `traces_replacing`, `llm_messages`) override `configure_insert` to pass `async_insert_busy_timeout_max_ms`. The value is read once at boot from `SPANS_CH_WAIT_FOR_ASYNC_INSERT_MS` (default 400) into `SPANS_CH_ASYNC_INSERT_BUSY_TIMEOUT_MAX_MS` in `app-server/src/ch/mod.rs` ÔÇö single env var covers all three tables since they share the same hot ingest path. Capping the max ceiling without touching durability (`wait_for_async_insert=1` still acks after disk flush). p50 client-observed wait drops from ~1 s to ~400 ms. Trade-off: ~2.5├Ś the parts/sec on these tables ÔÇö monitor `system.parts` active count for `spans` after rolling out and watch for `parts_to_delay_insert`/`parts_to_throw_insert` proximity (CH stock defaults are 1000 / 3000; `spans` has no `PARTITION BY` so it's all one virtual partition). Steady-state was ~179 active parts on `spans` pre-change, so the 2.5├Ś headroom still leaves >2├Ś margin to the delay threshold.
- Do NOT set this option on the global client ÔÇö it would also apply to non-batched, low-volume inserts (notifications, signal_events) where the larger coalescing window is harmless and the extra parts churn is wasted.

## RabbitMQ Connections

- Reconnect is delegated to lapin 4.x via `ConnectionProperties::enable_auto_recover()` (called in `app-server/src/main.rs`; the `mq` module is `mod.rs` + `rabbit.rs` + `tokio_mpsc.rs` + `utils.rs`). The same `Connection` handle stays alive across broker blips ÔÇö lapin redials TCP, re-opens channels, and re-establishes consumers in place. The `MessageQueue` trait exposes `is_healthy()` as a readiness probe.
- `get_receiver` in `app-server/src/mq/rabbit.rs` and the publisher channel pool's `create_channel()` are both wrapped in `tokio::time::timeout` (15s and 10s respectively). lapin can hang inside a setup call against a half-dead connection; the per-attempt timeout makes the outer backoff retry actually fire. Worker connect backoff caps at 5s `max_interval` (`worker/mod.rs`, `batch_worker/worker.rs`) so a transient pod-init failure resolves within seconds, not the 60s seen in the legacy supervisor era.
- Liveness (`/health`) is process-only and always returns 200 (`app-server/src/routes/probes.rs`). Readiness (`/ready`) gates on `MessageQueue::is_healthy()` AND `Cache::is_healthy()` and returns 503 when either is mid-reconnect ÔÇö so k8s steers traffic away during a blip but does NOT kill the pod. Do not re-couple `/health` to dependency state: a 3-replica cluster losing one node would otherwise crashloop the entire app-server fleet while the resilient connections are doing their job.
- Quorum queues are already declared (`x-queue-type=quorum` in `main.rs`) so message data survives node loss; the resilience work is purely about client-side reconnect, not durability.

## Redis Connections

- `redis::aio::MultiplexedConnection` does NOT auto-reconnect ÔÇö once the TCP socket drops, every command on that handle errors out forever. Wrap it in `ResilientRedisConnection` (`app-server/src/cache/connection.rs`), which tracks `is_connected` via an `AtomicBool`, runs a 30s `PING` health-check loop, listens for `notify_error()` from callers, and on real failure redials with uncapped exponential backoff and atomically swaps the new `MultiplexedConnection` in via `ArcSwap`. The supervisor PINGs first to coalesce transient hiccups before declaring the connection dead.
- Three timeouts bound every network call so a silent socket can't wedge the supervisor or hang command call sites: `DIAL_TIMEOUT` (10s) on `get_multiplexed_async_connection`, `PING_TIMEOUT` (5s) on the supervisor's coalesce check and the periodic health PING, and `COMMAND_RESPONSE_TIMEOUT` (10s) applied via `set_response_timeout` to the returned `MultiplexedConnection` so every cache get/set inherits the bound. Without these, a half-dead TCP connection could (a) hang process startup forever at pod init, (b) wedge the supervisor's `ping` before it reached the redial branch, or (c) hang the request that triggered any cache operation.
- One `ResilientRedisConnection` is shared by `RedisCache` and `RedisPubSub`'s publish path. Both call `connection.current_clone()` per operation (the `MultiplexedConnection` itself is a cheap-to-clone handle) and call `connection.notify_error()` on every command error. Pub/Sub's *subscribe* path still uses a dedicated `redis::aio::PubSub` connection from `client.get_async_pubsub()` ÔÇö Redis pins subscription state per socket, so the multiplexed connection can't be reused there. Subscribe is kept simple: when the stream ends, the loop exits and the caller decides whether to re-subscribe.
- Startup degrades gracefully: if the initial `ResilientRedisConnection::connect` fails, app-server falls back to `Cache::InMemory` and `PubSub::InMemory` (logged as a warning). Once connected, the supervisor keeps trying forever ÔÇö Redis flapping does not crashloop the pod.
- `Cache::is_healthy()` is the trait gate read by `/ready`. The Redis variant returns `connection.is_connected()` (the AtomicBool flipped by the health-check loop and the supervisor); the InMemory variant returns `true`.

## CSP and rrweb Browser Session Replay

- The site-wide CSP in `frontend/next.config.ts` deliberately omits `base-uri`. rrweb-player rebuilds captured DOM by injecting `<base href="<recorded-origin>">` so relative URLs in the snapshot resolve against the original site. `base-uri 'self'` blocks that and the replay loses every relative asset (LAM-1622). Do NOT add `base-uri` back without first moving the player into a CSP-relaxed iframe.
- The `Blocked script execution ... 'allow-scripts' is not set` console warnings during replay are NOT a bug ÔÇö rrweb-player intentionally creates its replay iframe with a sandbox that omits `allow-scripts` so captured `<script>` tags can't execute inside lmnr.ai. They render as warnings in dev tools but the replay itself is unaffected.

## Frontend Best Practices

### One component per file

Related components should be in a folder named by the parent component (`my-list/`) and the parent component should follow the index.tsx pattern (`my-list/index.tsx`) and all related components should be in the folder (`my-list/my-list-item.tsx`).

Please do your best to keep components <150 lines.

### Bias towards complex logic and state in the Zustand store

When you anticipate lots of complex state management with useState and useEffects, this would be a good time to rethink or refactor and move state into a shared store and expose derived state via selectors.

### Avoid syncing URL params with Zustand store antipattern

Use the nuqs library to handle url param state when possible. Avoid using a useEffect to sync URL param state with the Zustand store. Prefer keeping source of truth as the useQueryState and passing in necessary state as function params to the store when needed.

### Use Zustand shallow to avoid unnecessary rerenders

Pass shallow as the equality function to useStore when applicable. That way even with a new selector reference each render, Zustand compares the result shallowly and won't re-render if the contents are the same.

### AbortController

Use an `AbortController` to cancel in-flight `fetch` requests when a newer request supersedes them or the component/store state they'll update has moved on. Pass the controller's `signal` to `fetch`; the browser rejects the promise with an `AbortError` when aborted, so bail without touching state in the catch.

**Example ÔÇö per-store cancellation of pagination when the underlying query changes** (`dashboard-editor-store.tsx`):

```typescript
const createStore = (props) => {
  // Closure-scoped per store instance ÔÇö no cross-instance leak.
  let paginationAbortController: AbortController | null = null;

  return createStore((set, get) => ({
    executeQuery: async (projectId) => {
      // A fresh execute is about to replace page-0 data; any in-flight
      // pagination would splice stale rows onto it.
      if (paginationAbortController) {
        paginationAbortController.abort();
        paginationAbortController = null;
        set({ tableIsFetching: false });
      }
      // ...
    },

    fetchNextTablePage: async (projectId) => {
      // Abort any prior pagination (rapid scroll fires multiple times).
      paginationAbortController?.abort();
      const controller = new AbortController();
      paginationAbortController = controller;

      try {
        const response = await fetch(url, { signal: controller.signal, ... });
        // ... process result, functional set ...
      } catch {
        // Aborted ÔÇö whoever replaced us owns state. Don't reset flags here.
        if (controller.signal.aborted) return;
        set({ tableIsFetching: false });
      } finally {
        if (paginationAbortController === controller) {
          paginationAbortController = null;
        }
      }
    },
  }));
};
```

**Problems this solves / when to use:**

- **Race conditions where an older request overwrites newer state.** Classic example: user paginates (page 1 in-flight), then changes the filter (triggers a new page-0 fetch). Without aborting, page 1 resolves after page 0 and splices stale rows onto fresh data.
- **Wasted network + server work** when the result is no longer needed. Aborting tells the browser to drop the request, which can also cancel the server-side query if the backend respects it.
- **Rapid user actions** like repeated scrolls, debounce-escaped clicks, or typing into a search field ÔÇö only the latest request's result should land.

Prefer `AbortController` over hand-rolled "snapshot state at start, compare at resolve, discard if drifted" patterns ÔÇö it's the standard browser primitive and cancels the actual network request, not just its effect on state.

**Gotchas:**

- Don't reset loading flags (`isFetching`, `isLoading`, etc.) in the abort branch of the catch. The operation that aborted you is responsible for the next state ÔÇö resetting here would race with it.
- When aborting from a different action, the aborting action must handle any loading flag the aborted action left behind (see `executeQuery` above clearing `tableIsFetching`).
- In the `finally`, only null out the shared controller ref if it still points at the current controller ÔÇö otherwise a newer operation already replaced it and you'd be clobbering its handle.
- In the success path, use functional `set((state) => ...)` rather than a closure over `state.data` so you merge with the latest value.

### Error handling

**Client-side fetch calls** (in `"use client"` components): Always wrap `fetch` calls in `try/catch`. Check `res.ok` before using the response. On error, show a toast notification to the user via `useToast()`. Extract the error message from the response JSON when available, falling back to a generic message.

```typescript
try {
  const res = await fetch(`/api/projects/${projectId}/resource`, { method: "POST", body: JSON.stringify(data) });
  if (!res.ok) {
    const errMessage = await res.json().then((d) => d?.error).catch(() => null);
    toast({ variant: "destructive", title: errMessage ?? "Something went wrong" });
    return;
  }
  // handle success
} catch {
  toast({ variant: "destructive", title: "Something went wrong" });
}
```

**API route handlers** (`app/api/**/route.ts`): Wrap the handler body in `try/catch`. Distinguish `ZodError` (return 400 with `prettifyError()`) from other errors (return 500). Always return a JSON response with an `error` field.

```typescript
try {
  const result = await someAction(input);
  return Response.json(result);
} catch (error) {
  if (error instanceof ZodError) {
    return Response.json({ error: prettifyError(error) }, { status: 400 });
  }
  return Response.json({ error: error instanceof Error ? error.message : "Internal server error" }, { status: 500 });
}
```

**Server components** (`page.tsx`): Let database/fetch errors propagate to the nearest `error.tsx` error boundary ÔÇö do **not** catch them and convert to `notFound()`. Only use `try/catch` or `.catch()` when you need a specific fallback value for optional data. Use `notFound()` only for genuinely missing resources (i.e. when a query returns `null`/`undefined`).
