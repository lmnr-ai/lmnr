Here's everything we shipped in March.

## Alerts and reports

You can now get notified when signals fire — through both Slack and email.

**Alerts** let you configure per-signal notifications. When a signal detects an event (e.g. "agent got stuck in a loop"), Laminar pushes a notification to a Slack channel, to your email, or both. Each alert links directly to the trace so you can jump straight to the issue.

**Reports** deliver scheduled summaries of signal activity. Subscribe to daily or weekly email digests, or route them to a Slack channel. Reports aggregate recent signal events so you can track trends without watching the dashboard.

Both systems are unified — every alert and every report supports Slack and email as notification targets, so you pick the channels that fit your workflow.

## Full-text search with highlighted snippets

Search now works across the full text of span inputs and outputs. Type a query in the traces page and results come back with highlighted snippets showing the matching context, so you can tell at a glance which traces are relevant without opening each one.

Under the hood, we added a text preprocessing pipeline that normalizes data before indexing — fixing tokenization issues where escape sequences caused bad tokens. Search results include snippet counts per trace and auto-expand matching spans in the trace view.

## Cost heatmap in trace view

A new toggle in the trace timeline lets you switch to a cost heatmap view. Spans are rendered as a red gradient based on relative cost — the most expensive span is fully red, zero-cost spans are muted. This makes it immediately obvious where you're spending money in a long agent run, without reading through individual span details.

## Custom SQL columns and sorting on traces

The traces table now supports custom columns defined by arbitrary SQL expressions. Click "Columns → Custom column," write a ClickHouse expression (or let AI generate one), and the result appears as a new column in your traces table. Custom columns persist in your browser and are sent to the backend for server-side evaluation.

All applicable trace columns (timestamp, duration, cost, tokens) are now sortable as well. Click any column header to sort, and the sort state is preserved in the URL for sharing.

## Signals improvements

Signals got a major round of updates this month:

- **Redesigned signals page.** The list view now shows sparkline charts per signal, a card-based layout, and an empty-state onboarding flow.
- **Inline trigger creation.** Triggers are now created directly inside the signal sheet instead of a separate tab, making the setup flow much faster.
- **Realtime and batch processing modes.** Each signal trigger can now be set to realtime (processed immediately as traces arrive) or batch mode.
- **Per-signal sampling.** For high-volume workloads, you can set a sampling percentage (1–95%) on any signal. The budget is distributed across users proportionally to their trace volume using probabilistic sampling.
- **Signal events panel in trace view.** A new panel in the trace viewer shows all signal events for that trace, with tabs per signal and links to open the full signal page.
- **`search_in_spans` tool.** The signal processing agent can now search through span inputs and outputs for specific text instead of relying on the full payload, making extraction more precise and cheaper.
- **Idempotent signal runs.** Duplicate processing is now prevented at the infrastructure level.

## Span view improvements

The span detail view got a significant upgrade:

- **Overview tab** for LLM spans that summarizes recent input messages and full output.
- **Server-side preview generation** replaces client-side parsing. The backend now detects the provider format (OpenAI, Anthropic, Gemini, LangChain) and generates previews automatically, with an LLM fallback for unknown structures.
- **Root span input/output columns** on the traces table — the root span's input and output are now materialized during trace ingestion and shown directly in the table, no click required.
- **Cached token counts** now appear in the span stats tooltip.

## Self-hosted improvements

- **Keycloak authentication.** Keycloak is now supported as an OAuth provider for self-hosted deployments. Set three environment variables and a "Continue with Keycloak" button appears on the sign-in page.
- **Improved user invitations.** In self-hosted mode, admins can now invite users who don't have accounts yet. A pending invitation is created, and when the user signs up, they're automatically added to the workspace.
- **Hidden dummy email input.** When real auth providers (Google, GitHub, Keycloak, etc.) are configured, the default email input is hidden to avoid confusion.

## Playground updates

- **Gemini 3 models** added to the playground: `gemini-3-flash-preview`, `gemini-3-pro`, and `gemini-3.1-flash-lite-preview`, all with thinking/budget configuration support.

## Python SDK

- **Streaming token usage** for Anthropic is now always reported, not just when `enrich_token_usage` is enabled. Cumulative usage from `message_delta` events is handled correctly.
- **OpenRouter model names** are no longer stripped of their provider prefix — `openai/gpt-4o` stays as `openai/gpt-4o` in span attributes.
- **`opentelemetry-instrumentation-threading`** is now a default dependency for better async trace context propagation.
